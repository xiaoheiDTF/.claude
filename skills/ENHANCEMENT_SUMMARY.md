### 4. code-implementer（代码实现）

**新增能力**：
- ✅ **增量修复模式** — "修一个、跑一次、确认无新问题"的 Build Fix 集成
- ✅ **中断恢复协议** — 读取检查点 → 确认状态 → 从断点继续
- ✅ **用户反馈机制** — 定位问题 + 最小修改 + 提炼偏好 + 更新 CLAUDE.md

**改动**：+122 行（重构）

**增量修复流程**：
```
1. 跑完整检查，拿到所有错误
2. 按文件/严重程度分组
3. 一次只修一个错误，修完立即重跑
4. 最多重试 3 次，仍失败则停下报告
```

### 5. code-tester（代码测试）

**新增能力**：
- ✅ **TDD 先行模式** — 先写测试 → 确认失败 → 提示实现 → 复跑验证
- ✅ **覆盖率门禁** — 检查阈值（默认 80%），未达标则补充测试
- ✅ **用户反馈机制** — 定位问题 + 局部修正 + 重新运行 + 记录偏好

**改动**：+104 行（重构）

**TDD 模式流程**：
```
1. 先分析接口（只看签名，不看实现）
2. 先生成测试（此时必然失败）
3. 运行测试确认失败（红灯）
4. 提示用户实现
5. 用户实现后复跑（绿灯）
```

## 核心设计理念

### 1. 质量门禁（Quality Gates）

每个技能在关键节点嵌入可执行的质量检查：
- **task-breakdown**: 验收标准必须可执行
- **code-reuse-finder**: 复用风险四维度检查
- **impl-planner**: 6 项自检清单
- **code-implementer**: 每步必检（格式化 → 静态检查 → 类型检查）
- **code-tester**: 覆盖率门禁（默认 80%）
- **web-research**: 时效性门禁 + 信息缺口识别

### 2. 中断恢复（Session Continuity）

支持长任务中断后无缝恢复：
- **impl-planner**: 恢复检查点（最后完成步骤 + 进行中步骤 + 待执行步骤）
- **code-implementer**: 读取检查点 → 确认状态 → 从断点继续
- **trace-call-chain**: 会话交接摘要（追踪结果 + 关键文件 + 建议下一步）

### 3. 用户反馈机制（Feedback Loop）

所有技能统一实现反馈优化循环：
```
用户不满意 → 定位问题 → 局部修正 → 重新验证
→ 若重复 2 次以上 → 主动建议 /learn 修正 <skill> <描述>
→ 固化到 Skill 规范或个人偏好库
```
