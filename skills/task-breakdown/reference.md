# Task Breakdown 模板与示例参考

本文件包含 task-breakdown Skill 的所有模板和示例，供 SKILL.md 按需引用。

---

## README.md 模板

```markdown
# 需求拆解报告

> 生成时间: YYYY-MM-DD HH:mm
> 原始需求: <用户描述的需求>

---

## 目录

| 文件 | 内容 |
|------|------|
| 本文件 | 速览卡 + 模块总览 + 依赖图 + 实现路线 |
| [M0-<简称>.md](./M0-<简称>.md) | <模块〇名称> — <一句话职责> |
| [M1-<简称>.md](./M1-<简称>.md) | <模块一名称> — <一句话职责> |
| ... | ... |
| [SUMMARY.md](./SUMMARY.md) | 功能总览表 + 建议与补全 |

---

## 速览卡

**核心目标**: <一句话>

**规模**: X 个模块 / X 个功能单元

**关键约束**:
- <约束 1>
- <约束 2>
- <约束 3>

---

## 模块总览

| 模块 | 职责 | 功能数 | 核心依赖 | 详情 |
|------|------|--------|---------|------|
| 〇、<名称> | <一句话> | X | 无 | [→ 查看](./M0-<简称>.md) |
| 一、<名称> | <一句话> | X | 〇 | [→ 查看](./M1-<简称>.md) |
| ... | ... | ... | ... | ... |

---

## 依赖关系图

\```mermaid
graph TD
    M0[〇、<名称>] --> M1[一、<名称>]
    M0 --> M2[二、<名称>]
    M1 --> M3[三、<名称>]
    M2 --> M3
\```

---

## 实现路线

| 批次 | 功能 | 说明 |
|------|------|------|
| P0 | <模块编号>.<功能编号> | 无依赖，最先实现 |
| P1 | ... | 依赖 P0 |
| P2 | ... | 依赖 P1 |

详细实现顺序（含各功能单元编号）见 [SUMMARY.md](./SUMMARY.md)。
```

**当拆解目标涉及与已有模块对齐时**，README.md 中必须增加以下段落（放在"实现路线"之后）：

```markdown
## 映射对照表

> 以下展示新模块与已有模块的功能对应关系，便于下游 Skill 快速定位可复用代码。

| 新模块 | 新功能单元 | 对应已有模块 | 对应已有功能 | 复用方式 |
|--------|----------|------------|------------|---------|
| M0-xxx | 0.1 功能名 | cdp/lifecycle/ | spawnBrowser() | 改写（API不同但逻辑同） |
| M0-xxx | 0.4 功能名 | cdp/lifecycle/ | detectBrowserProcess() | 直接复用（与API无关） |
| M7-xxx | 7.1 功能名 | cdp/state/types.ts | BrowserState 接口 | 直接复用（共享类型定义） |
```

## 模块详情文件模板

每个模块一个文件，包含该模块所有功能单元的完整信息。文件命名为 `M0-xxx.md` ~ `Mn-xxx.md`。

```markdown
# <模块编号>、<模块名称>

> 目标: <一句话描述这个模块要完成什么>

---

## 功能单元清单

| # | 功能单元 | 说明 | 验收标准 | 依赖 |
|---|---------|------|---------|------|
| <编号>.1 | <名称> | <做什么> | <怎样算完成> | 无 |
| <编号>.2 | <名称> | <做什么> | <怎样算完成> | <编号>.1 |

---

## 功能单元详情

### <编号>.1 <功能单元名称>

- **描述**: <做什么，输入什么，输出什么>
验收: <怎样算完成>
- **依赖**: 无

---

### <编号>.2 <功能单元名称>

- **描述**: <做什么，输入什么，输出什么>
验收: <怎样算完成>
- **依赖**: <编号>.1

<details>
<summary>补充说明</summary>

- <只在需要时出现，如跨平台策略、限制说明、技术细节等>
- <没有补充说明的功能单元不需要此折叠块>

</details>

---
```

**注意**：
- 功能单元清单表格提供快速扫读视图
- 功能单元详情逐个展开，用 `---` 分隔
- 只有需要补充说明（如跨平台策略、限制、前置条件等）的功能单元才用 `<details>` 折叠块
- 不要在模块文件中重复依赖关系图和实现路线，这些只在 README.md 中出现一次

## manifest.json 模板

```json
{
  "skill": "task-breakdown",
  "version": "1.0",
  "generated_at": "YYYY-MM-DD HH:mm",
  "requirement": "<原始需求描述>",
  "modules": [
    { "id": "M0", "name": "<模块名称>", "short": "<模块简称>", "file": "M0-<简称>.md", "units_count": X }
  ],
  "units_count": X,
  "languages": ["typescript", "python"],
  "directories": [
    { "module": "M0", "path": "<相对项目根目录的目标路径>" }
  ]
}
```

**用途**：下游 Skill（code-reuse-finder、impl-planner）读取此文件，自动获取拆解报告的范围、模块、涉及语言和目标目录，无需解析 Markdown。

## 拆解示例

需求："用户点击按钮后自动填写表单并提交"

**好的拆解**（控制在合理范围）：

| # | 功能单元 | 说明 | 验收标准 | 依赖 |
|---|---------|------|---------|------|
| 1.1 | 识别目标表单 | 根据意图定位页面上的目标表单 | 能从页面中找到正确的表单元素 | 无 |
| 1.2 | 提取表单结构 | 分析表单的字段、类型、必填项 | 输出完整的字段清单 | 1.1 |
| 1.3 | 生成填写内容 | 为每个字段生成对应值 | 每个必填字段都有对应值 | 1.2 |
| 1.4 | 执行填写并提交 | 填入值 → 校验 → 提交 | 提交成功并获得反馈 | 1.3 |

**坏的拆解**（越界发散）：
- 自行拆出"支持多种表单框架（Ant Design、Element UI）"
- 自行拆出"表单填写历史记录"
- 自行拆出"OCR 识别验证码"
（这些都是"可能需要"但用户没要求的功能）
