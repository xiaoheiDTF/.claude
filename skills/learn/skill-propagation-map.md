# 模式→Skill 反向传播映射表

> learn 的 L3 资源文件。沉淀模式后，按此表分析哪些 Skill 应该受益。

---

## 一、传播分析三步法

### Step 1: 判断传播类型

| 模式特征 | 传播类型 | 说明 |
|---------|---------|------|
| 模式揭示了某个 Skill 的工作流缺陷 | **流程修补** | 在 Skill 的工作流程中增加步骤或检查点 |
| 模式提供了通用的防御性规则 | **规则追加** | 在 Skill 的关键规则中追加条目 |
| 模式暴露了质量检查盲区 | **门禁加固** | 在 Skill 的质量门禁/验收标准中补充检查项 |
| 模式提供了避坑经验 | **避坑警示** | 在 Skill 的核心原则或关键规则中追加警告 |
| 模式与现有 Skill 无直接关系 | **不传播** | 仅保留在模式库中 |

### Step 2: 定位受影响的 Skill

用以下关键词匹配表快速定位：

| 模式关键词 | 受影响 Skill | 更新目标 section |
|-----------|-------------|-----------------|
| 需求拆解、粒度、模块划分、验收标准 | task-breakdown | 核心原则 / 关键规则 / 质量门禁 |
| 复用、现有代码、重复实现、已有实现 | code-reuse-finder | 判断标准 / 关键规则 / 复用风险提示 |
| 执行计划、步骤顺序、依赖关系、实现边界 | impl-planner | 核心原则 / 关键规则 / 计划质量门禁 |
| 代码实现、编码、修改边界、增量修复 | code-implementer | 核心原则 / 关键规则 |
| 测试、断言、覆盖率、测试策略、mock | code-tester | 关键规则 / 作用域边界规则 |
| 调用链、调试、定位、追踪、根因 | trace-call-chain | 关键规则 |
| 提交、commit、分支、push、PR | git-push | Commit Message 规范 / 关键规则 |
| 调研、搜索、文档、信息源 | web-research | 核心原则 / 关键规则 |

### Step 3: 生成 correction 条目

将分析结果写入 `corrections.md`，格式：

```markdown
## 对 <skill名> 的修正

> 记录于 YYYY-MM-DD
> **来源**: 模式库反向传播 — <模式名称> (<模式库文件名>)

**问题**: <该 Skill 当前缺少什么，可能导致什么问题>
**期望行为**: <加入此经验后 Skill 应该怎么做>
**修正建议**: <具体改哪个 section，加什么内容>
**传播类型**: 流程修补 / 规则追加 / 门禁加固 / 避坑警示
**已修正**: 否
```

## 二、模式类别→Skill 传播矩阵

以下矩阵列出每种模式类别**最可能**影响的 Skill 和注入位置。

### 调试排错模式 (debug.md) 的传播路径

| 典型发现 | 传播到 | 注入位置 | 更新内容举例 |
|---------|--------|---------|------------|
| 排查路径不够高效 | trace-call-chain | 关键规则 | 追加排查策略规则 |
| 测试断言掩盖了真实问题 | code-tester | 关键规则 | "断言失败时先确认根因" |

### 架构设计模式 (architecture.md) 的传播路径

| 典型发现 | 传播到 | 注入位置 | 更新内容举例 |
|---------|--------|---------|------------|
| 接口设计缺参数（如路径硬编码） | impl-planner | 核心原则 | "设计接口时禁止硬编码可变路径" |
| 模块边界不清晰 | task-breakdown | 关键规则 | "拆解时必须画清模块边界" |
| 设计决策没有记录 | impl-planner | 关键规则 | "每个步骤标注设计决策理由" |

### 重构手法模式 (refactor.md) 的传播路径

| 典型发现 | 传播到 | 注入位置 | 更新内容举例 |
|---------|--------|---------|------------|
| 重构前缺少测试保护 | code-tester | 关键规则 | "重构目标必须先有测试覆盖" |
| 重构粒度过大导致回退困难 | code-implementer | 关键规则 | "重构每步保持可编译" |
| 重构改变了公共接口 | impl-planner | 核心原则 | "重构计划中标注破坏性变更" |

### 代码片段模式 (snippets.md) 的传播路径

| 典型发现 | 传播到 | 注入位置 | 更新内容举例 |
|---------|--------|---------|------------|

| 发现了更好的错误处理模式 | code-implementer | 关键规则 | "错误处理使用 X 模式" |

## 三、传播质量检查

生成 correction 前，逐项确认：

- [ ] 修正建议是否**具体到 section 名**（"追加到关键规则"而非"改一下"）
- [ ] 修正内容是否**用通用语言**（不含项目专有名词）
- [ ] 是否**不与目标 Skill 现有规则矛盾**（先读再写）
- [ ] 是否**可独立执行**（不依赖本次对话的上下文才能理解）
- [ ] 修正后目标 Skill 是否**仍在 500 行以内**

## 四、不传播的判断标准

以下情况模式只保留在模式库，不产生 correction：

1. **纯项目专属知识** — 标记了 `[项目专属]` 的模式
2. **过于具体的技巧** — 仅适用于某个库/框架的特定版本
3. **已有等价规则** — 目标 Skill 中已存在覆盖此场景的规则
4. **经验不够成熟** — 只出现过一次，尚未在第二个场景验证
