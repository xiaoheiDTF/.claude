## 测试失败根因定位 — 沿调用链区分"测试问题"还是"源码限制"

> 沉淀于 2026-04-12，来源：lifecycle E2E 测试 3/8 失败，排查 Network.getCookies 错误

**通用场景**: 集成/E2E 测试报错时，需要区分是"测试写错了"还是"被测代码的设计限制"
**识别信号**: 测试断言失败但错误信息来自被测系统内部（如协议错误、底层 API 不可用）；改断言能让测试通过但可能掩盖问题
**通用做法**:
  1. 从测试错误信息出发，沿调用链逐层读源码（测试 → 被测接口 → 内部实现 → 底层依赖）
  2. 定位到根因后判断类别：测试前置条件不满足 / 源码对异常无容错 / 底层 API 在特定环境不可用
  3. 如果是源码限制：调整测试策略绕过（用替代 API、改变验证方式），不为测试改源码
  4. 如果是测试问题：修正测试的前置条件或断言
**原因**: 表面看是测试断言失败，实际根因可能在调用链的任何一层。盲目改断言可能掩盖真实 bug
**避坑**: 不要看到错误就改测试断言，先读完调用链确认根因。"让测试通过"不等于"问题解决了"
**适用举例**: 数据库测试报连接错误（环境问题 vs 连接池 bug）、API 测试报 403（权限配置 vs 中间件 bug）、UI 测试元素找不到（渲染时序 vs 组件 bug）
