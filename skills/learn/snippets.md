## 缺失构建工具时的质量门禁降级策略

> 沉淀于 2026-04-16，来源：在 Windows 环境为 travel-agent-java 写代码时，本机未安装 mvn/javac

**通用场景**: 在临时环境/新机器/容器外执行代码实现任务，项目构建链（maven、gradle、eslint、tsc 等）未就绪
**识别信号**: 执行 `mvn compile` 或 `eslint` 时返回 `command not found`
**通用做法**:
1. 尝试定位工具（`where mvn`、`javac -version`）
2. 确认缺失后，用以下方式保证质量不裸奔：
   - 人工检查明显的语法错误（括号匹配、导入语句、泛型一致性）
   - 检查接口实现关系的完整性
   - 验证包路径与文件路径的一致性
3. 在实现报告和日志中明确标注"质量门禁已跳过（原因：环境缺少 X）"
4. 给出用户可执行的后续验证命令
**原因**: 硬等环境配置会阻塞交付；但跳过检查必须有记录和补救方案，否则后续集成时问题会集中爆发
**避坑**: 不要假装"已经跑过检查了"；明确标注跳过并给出后续验证命令是对用户负责
**适用举例**:
- 远程环境只有 Python 但没有 Node，需要改前端代码
- 新机器没有安装 Go 工具链但需要写 Go 服务
- Docker 容器内缺少数据库迁移工具但需要生成 SQL

## 测试用例只增不改原则（测试公平性保障）

> 沉淀于 2026-04-16，来源：用户在编写 parser 框架流式输出测试时提出的原则

**通用场景**: 编写和维护测试用例时，为保证测试作为独立验证手段的公正性，所有新写的测试用例只可以去添加（新增 @Test 方法），不可以修改已有的测试用例
**识别信号**:
- 发现 bug 后下意识想"改一下测试让它通过"
- 测试失败时第一反应是调断言值而不是查根因
- 重构后想"删掉这个不方便的测试用例"
**通用做法**:
1. 测试用例写好后即冻结，后续只允许新增测试方法
2. 如果已有测试的预期值确实有误（测试本身写错了），需用户明确确认后才可修正
3. 发现 bug 后应新增测试覆盖 bug 场景，而非修改旧测试的断言
4. 重构后旧测试仍必须通过——如果过不了，说明重构改变了行为契约
**原因**: 修改已有测试会削弱检查力，等于"改答案来适应错误"，违背测试作为独立验证手段的根本目的
**避坑**: "测试本来就不对"是修改的合理理由，但必须由人确认，不能由 AI 自行判断
**适用举例**:
- 单元测试补充新场景时，不改动已有断言
- 集成测试发现边界行为与预期不同时，新增测试覆盖而非修改旧测试
- 安全测试中发现新攻击面时，新增测试用例而非改写已有
- TDD 红灯阶段写好的测试，绿灯阶段不允许修改测试只允许修改实现
