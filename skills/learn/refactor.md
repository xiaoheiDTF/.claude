# 重构手法模式库

## 执行计划的局部热更新（不重新生成整个计划）

> 沉淀于 2026-04-16，来源：用户在使用 03-plan 执行计划过程中提出"实现层放入子包"的结构调整需求

**通用场景**: 已生成的技术方案/文档需要响应小幅结构变更（包路径调整、命名规范变更、新增子目录），但功能和验收标准不变
**识别信号**: 用户说"把 X 放到 Y"、"X 应该叫 Y"、"加个 Z 子目录"——属于结构性调整而非功能变更
**通用做法**:
1. 不重新生成整个计划（避免破坏原有依赖链和验收场景）
2. 定位所有受影响的位置：README.md 文件结构图、manifest.json target 路径、各 Sxx.md 步骤详情
3. 同步更新预创建目录和对应 CLAUDE.md
4. 保持步骤编号、依赖关系、验收场景不变
5. 重跑验证脚本确认计划完整性
**原因**: 全量重写计划成本高且容易引入新的不一致；局部更新能保持原有验收标准和依赖链的稳定性
**避坑**: 不要只改 README.md 而忘记改 manifest.json，否则下游 code-implementer 会读错目标路径
**适用举例**:
- 接口文档生成后，用户要求按 Controller 再分子包
- 数据库迁移脚本已生成，用户要求按模块分目录
- API 契约已定，用户要求 DTO 统一放到 `dto/request/` 和 `dto/response/`
