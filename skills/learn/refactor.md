## 大规模批量规范化 — 分步执行 + Todo 追踪

> 沉淀于 2026-04-13，来源：全项目 CLAUDE.md 缺失/不完整扫描与修复，涉及 30 个目录

**通用场景**: 项目已有大量目录缺少规范文档/配置文件时，需要批量生成和补全
**识别信号**: 扫描发现 10+ 个缺失或不完整的文件，直接逐个处理容易遗漏或重复
**通用做法**:
  1. **先全量扫描** — 用检查模式获取完整的缺失/不完整/过时列表
  2. **分类建 Todo** — 按操作类型分组（生成缺失、修复过时、补全不完整、初始化索引），每组一个 Todo 项
  3. **按顺序执行** — 生成缺失 → 修复过时 → 补全不完整 → 初始化 registry → 生成索引
  4. **内容定制** — 每类文件根据上下文定制内容（引用语言规则、匹配目录职责），不套用统一模板
  5. **最终报告** — 输出统计和详细列表，确认无遗漏
**原因**: 大规模操作如果逐个处理容易丢失进度和遗漏文件。分类分步执行让每步可验证、可回退
**避坑**: 不要为所有文件套用同一个模板 — 不同目录的职责不同，代码规范和 Review 检查清单应该针对性编写
**适用举例**: 批量添加 ESLint 配置到多个子项目、批量补充 API 文档、批量更新依赖版本并同步 lockfile、批量添加 .gitignore 规则

<details>
<summary>原始案例</summary>

**项目场景**: zwyAi 项目 30 个含代码目录中，4 个缺失 CLAUDE.md、1 个目录结构过时、25 个缺少"代码规范"和"Review 检查清单"段落、module-registry 完全不存在
**具体做法**: 建立六步 Todo（生成缺失 → 修复过时 → 补全不完整 → 初始化 registry → 生成索引 → 输出报告），每步完成后标记 completed。每个 CLAUDE.md 根据目录职责从 .claude/rules/typescript.md 中选取 3-5 条关键约束写入代码规范段落
</details>

## 系统级魔法变量扫描应覆盖「全栈 + 分优先级」

> 沉淀于 2026-04-14，来源：对 travel-web + travel-agent-java 的全栈魔法值扫描

**通用场景**: 代码 review 或技术债务清理时，需要找出应该抽取为常量/配置的硬编码值
**识别信号**: 同一数字/字符串在 3 个以上文件出现；配置文件中缺少对应项；文档与代码中的值不一致
**通用做法**:
  1. 不要只扫单端，应同时扫描前端、后端、配置文件
  2. 用正则/Grep 找出重复出现的字面量，重点关注：超时时间、端口地址、协议字段名、业务阈值（如截断长度）、角色/状态字符串
  3. 按影响范围分优先级：
     - 高优先级：环境相关（端口、代理地址）、性能相关（超时、截断 token 数）、协议相关（event 名、JSON 字段名）
     - 中优先级：错误码、角色枚举字符串
     - 低优先级：单处使用的错误提示文案
**原因**: 魔法变量是技术债务中最容易累积、最难在运行时发现的类型。全栈扫描能暴露跨端不一致（如两端超时时间不同）
**避坑**: 不要试图一次性抽完所有字面量，优先处理「改一处忘改另一处就会出 bug」的那些
**适用举例**: 微服务中各服务超时时间不一致、移动端和 Web 端 API base URL 硬编码不同、算法服务中截断阈值散落各处

<details>
<summary>原始案例</summary>

**项目场景**: 旅游 Agent 全栈项目（Vue 前端 + Spring Boot 后端）中扫描魔法值
**具体做法**: 同步扫描 `travel-web/src` 和 `travel-agent-java/src/main`，整理出 30+ 处硬编码。高优先级包括：`60_000L` SSE 超时、`4000` 历史截断 token 数、Vite 代理 `5173`/`localhost:8000`、SSE 事件名和 JSON 字段名（`token`/`intro`/`done`/`content` 等）。按 🔴高/🟡中/🟢低 三档优先级输出修复建议清单
</details>
