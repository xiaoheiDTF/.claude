## 对 task-breakdown 的修正 — 需求分析后增加模块设计对齐 + 方案先行沟通

> 记录于 2026-04-13
> **来源**: 模式库反向传播 — 需求分析后模块设计对齐 + 方案先行沟通 (architecture.md)

**问题**: task-breakdown 拆解完需求后直接输出报告，缺少两步：①对照现有目录/模块结构确认拆解是否合理；②向用户展示方案概要（"我打算分这几个模块来做"）让用户确认
**期望行为**: 拆解完成后，增加"模块设计对齐"步骤（检查现有目录结构、确认新模块位置），并在最终输出前先展示方案概要供用户确认
**修正建议**: 
  - 在"第五步：生成报告文件"前，增加"第四步半：模块设计对齐"——读取项目现有目录结构，确认每个功能单元的目标位置，标注哪些目录需要新建、哪些 CLAUDE.md 需要更新
  - 在输出拆解结果前，先用自然语言告诉用户"我打算分 X 个模块来完成，分别是……，每个模块负责……"，等用户确认后再生成文件
**传播类型**: 流程修补
**已修正**: 是 (2026-04-13)

## 对 task-breakdown 的修正 — 需求阶段充分理解用户需求

> 记录于 2026-04-13
> **来源**: 模式库反向传播 — 方案先行沟通 (architecture.md)

**问题**: 用户说"只需要做聊天部分"，但拆解时未充分确认"聊天部分"的边界（是否含 Java 后端？动态调整？卡片渲染？），导致后续需要用户反复修正
**期望行为**: 需求分析阶段先确认功能边界和优先级，然后告诉用户"我理解你需要的是这些功能，分这几个模块"，确认后再拆解
**修正建议**: 
  - 在"第二步：需求分析"中增加"边界确认"子步骤：列出用户提到的功能点 + 可能相关的功能点，让用户确认哪些在范围内、哪些不在
  - 确认后输出一句话的功能范围描述，用户认可后再继续拆解
**传播类型**: 流程修补
**已修正**: 是 (2026-04-13)

## 对 impl-planner 的修正 — 技术栈变更时级联更新已有文档

> 记录于 2026-04-13
> **来源**: 模式库反向传播 — 技术栈变更级联更新 (architecture.md)

**问题**: 用户在计划阶段要求变更技术栈时，impl-planner 只重写计划文件，不检查和更新已生成的 CLAUDE.md 文件，导致子目录规范文件引用旧技术栈（旧 API 路径、旧函数签名、旧数据结构）
**期望行为**: 技术栈变更后，除了重写计划文件，还必须扫描所有受影响的 CLAUDE.md 并更新
**修正建议**: 
  - 在"第二步：交叉分析"后增加检查：如果本次计划的技术栈与已有 CLAUDE.md 描述不一致，列出所有需更新的 CLAUDE.md 文件
  - 在"第十步：预创建目录和模块 CLAUDE.md"中增加"已有 CLAUDE.md 内容校验"——检查已存在文件是否与新技术栈一致，不一致则标记为需更新
  - 在质量门禁中增加检查项："所有受影响目录的 CLAUDE.md 是否已更新为新技术栈"
**传播类型**: 门禁加固 + 流程修补
**已修正**: 是 (2026-04-13)

## 对 code-implementer 的修正 — 最后一步强制交付物扫描

> 记录于 2026-04-13
> **来源**: 模式库反向传播 — 实现交付物检查清单 + 交付物对照扫描 (architecture.md)

**传播类型**: 流程修补 + 门禁加固

## 对 code-implementer 的修正 — 技术栈变更后同步 CLAUDE.md

> 记录于 2026-04-13
> **来源**: 模式库反向传播 — 技术栈变更级联更新 (architecture.md)

**问题**: 技术栈变更后，code-implementer 在"第四步：同步模块 CLAUDE.md"中只检查新建目录，不更新已有目录中过时的 CLAUDE.md，导致规范文件引用旧技术栈
**期望行为**: 同步 CLAUDE.md 时，不仅检查新建目录，还要校验已有目录的 CLAUDE.md 是否与当前技术栈一致
**修正建议**: 
  - 在"第四步：同步模块 CLAUDE.md"中，增加"已有 CLAUDE.md 内容校验"：
    - 读取每个受影响目录的 CLAUDE.md
    - 检查是否引用了旧技术栈关键词（如旧框架名、旧 API 路径、旧函数签名）
    - 不一致的标记为需更新，按模板重写
  - 在关键规则中追加："更新 CLAUDE.md 时不仅看新建目录，也要校验已有目录的内容是否过时"
**传播类型**: 规则追加 + 门禁加固
**已修正**: 是 (2026-04-13)
