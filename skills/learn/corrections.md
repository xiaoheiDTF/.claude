> 记录于 2026-04-13
> **来源**: 模式库反向传播 — 验证脚本传参错误导致产物生成在错误位置 (debug.md)

**问题**: code-tester 的关键规则第 18 条写的是"bash $CLAUDE_SKILL_DIR/check-deliverables.sh <子项目根目录的绝对路径>"，但 check-deliverables.sh 用 `find -maxdepth 1` 在传入目录直接查找测试文件。传入子项目根目录会导致找不到测试文件，然后为了通过检查在根目录错误生成配套文件，污染子项目结构

  - 在关键规则中追加避坑："check-deliverables.sh 的参数必须指向包含 .test.* 文件的目录，不要传子项目根目录"

## 对 task-breakdown 的修正 — 验证脚本 JSON 校验依赖 Python 缺少 fallback

> 记录于 2026-04-13
> **来源**: 会话回顾 — validate-breakdown.sh 用 python -m json.tool 验证 JSON，但 Windows 环境可能只有 Store 占位符无实际 Python

**问题**: validate-breakdown.sh 第 100 行硬编码 `python -m json.tool` 做 JSON 格式验证，但 Windows 环境中 `python` 可能只是 Microsoft Store 占位符（exit code 49），导致格式正确的 manifest.json 也报「JSON 格式错误」
**期望行为**: JSON 验证有 fallback 机制，Python 不可用时自动降级到 Node.js（项目必有 Node 环境）
**修正建议**: 
  - 在 validate-breakdown.sh 中将 JSON 验证改为：先尝试 `python -m json.tool`，失败则 fallback 到 `node -e "JSON.parse(require('fs').readFileSync('$FILE','utf8'))"`
  - 或直接使用 Node.js（前端项目必有）作为 JSON 验证工具
**传播类型**: 流程修补
**已修正**: 是 (2026-04-13)
