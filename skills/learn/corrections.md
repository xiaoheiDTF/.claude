## 对 code-implementer 的修正

> 记录于 2026-04-15
> 来源: 模式库反向传播 — "魔法变量修复三层工作流" (refactor.md)

**问题**: code-implementer 已有"魔法字符串三级治理"规则（判定何时提取），但缺少具体的实现工作流——当计划涉及魔法变量修复时，实现者不知道常量/枚举/配置的标准写法模板，也不知道 Java switch-case 避坑。
**期望行为**: 遇到魔法变量修复步骤时，按标准模板写常量/枚举/配置，消费层替换遵循"纯机械替换"原则。
**修正建议**: 在关键规则中追加以下内容：
1. **常量类模板（Java）**: `public final class` + `private` 构造函数 + `public static final String`
2. **枚举模板（Java）**: `@JsonValue` + `fromValue()` 反查方法
3. **配置模板（Java）**: Properties 内部类 + getter/setter + YAML 环境变量覆盖
4. **常量文件模板（JS）**: 全部 `export const UPPER_SNAKE_CASE`，按域分组，禁止 `export default`
5. **switch 避坑（Java）**: `enum.getValue()` 不是编译期常量，不能用于 switch，改用 if-else + `equals()`
6. **替换原则**: 纯机械替换、值完全一致、不改接口签名。
注意：这些写法模板已写入 `.claude/rules/java.md` 和 `.claude/rules/javascript.md` 的"魔法变量治理"段落，code-implementer 执行时会自动加载这些 rules。Skill 层面只需在"工作流第二步：按步骤逐一实现"中加一句提醒："如果步骤涉及常量/枚举/配置创建，参考 `.claude/rules/<语言>.md` 中的魔法变量治理段落"
**传播类型**: 规则追加
**已修正**: 是 (2026-04-15) — 已写入 code-implementer SKILL.md 第二步第 3 点

## 对 code-implementer 的修正

> 记录于 2026-04-15
> 来源: 用户反馈 — "code-implementer 生成的业务代码缺少日志语句，调试时需手动补充"

**问题**: code-implementer 只关注功能逻辑的正确性，不主动在业务代码中添加日志语句。导致调试时缺少关键信息（入口参数、出口结果、异常详情、状态变更），需要用户手动补 log。
**期望行为**: 实现业务代码时，在以下四类场景自动添加日志：
1. **关键业务节点** — Service 层入口/出口（记录入参和返回值）、阶段转换、外部调用前后
2. **异常处理** — catch 块中必须 `log.error`，包含异常堆栈（`log.error("msg", e)` 而非 `log.error(e.getMessage())`）
3. **状态变更** — 会话创建/删除、阶段转换、配置值变更等关键状态变化时记录变更前后值
4. **外部交互** — HTTP 请求/响应（URL、状态码、耗时）、SSE 事件发送、数据库读写等与外部系统的交互
**修正建议**: 在 code-implementer SKILL.md "关键规则"中追加一条"业务代码必须包含日志"规则，并在第二步"按步骤逐一实现"中添加日志自检。
**传播类型**: 规则追加
**已修正**: 是 (2026-04-15) — 已写入 code-implementer SKILL.md "关键规则"业务代码必须包含日志"及第二步日志自检
