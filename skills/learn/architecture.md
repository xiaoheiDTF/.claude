## 路径硬编码导致检测函数静默失效

> 沉淀于 2026-04-12，来源：E2E 场景4 两个浏览器共用自定义 userDataDir 时锁检测失败

**通用场景**: 检测/校验函数内部硬编码默认路径，而调用者使用自定义路径时，检测逻辑静默失效（永远返回"正常"）
**识别信号**: 函数签名不接受路径参数；内部直接拼接默认路径常量；检测结果永远为"无冲突"但下游报超时/崩溃
**通用做法**:
  1. 检测函数必须接受实际路径作为参数，不可内部硬编码
  2. 若无法改函数签名，调用侧必须补充独立的检测逻辑
  3. 测试必须覆盖"默认路径"和"自定义路径"两种情况
  4. 当检测函数无法覆盖自定义路径时，接受下游系统自身的错误报告（不同错误信息）
**原因**: 硬编码路径让检测函数的覆盖范围窄于实际使用范围，形成"检测盲区"。下游系统（如外部进程）可能有自己的检测机制，但报错形式不同
**避坑**: 不要假设"检测函数返回正常 = 真的正常"。当使用非默认配置时，必须验证检测函数是否覆盖了实际路径
**适用举例**: 文件锁检测、缓存目录校验、配置文件 discovery、健康检查探针路径、日志目录存在性检查

<details>
<summary>原始案例</summary>

**项目场景**: `checkProfileLock()` 硬编码查默认 Profile 目录的 `SingletonLock` 文件。使用自定义 `userDataDir` 启动 Chrome 时，锁文件在自定义目录，函数找不到锁。Chrome 自行检测后以 exit code 21 退出，表现为端口超时而非"浏览器已在运行"
**具体做法**: E2E 测试应接受两种错误信息（锁冲突或超时）；若要根治需修改 `checkProfileLock` 函数签名，传入实际 Profile 路径
</details>

## 技术栈变更级联更新

> 沉淀于 2026-04-13，来源：旅游 Agent 项目中途将 httpx 换 LangChain、HTML 换 Vue 3，导致已生成的 CLAUDE.md、目录结构、执行计划全部过时

**通用场景**: 项目中途变更技术选型（框架、库、语言），已生成的文档、规范文件、目录结构、配置文件未同步更新，导致后续开发按过时信息执行
**识别信号**: 用户说"换成 X 框架"/"改用 Y 方案"；技术栈变量发生变化（语言、框架、依赖库、通信协议）
**通用做法**:
  1. 变更确认后，**立即列出受影响文件清单**（CLAUDE.md、配置文件、执行计划、目录结构）
  2. 将受影响文件分为"需重写"/"需更新段落"/"需删除"三类
  3. 逐一更新，每更新一个确认与新技术栈一致
  4. 清理不再需要的旧目录和旧文件
  5. 验证：抽查 3 个文件确认内容与新技术栈一致
**原因**: 技术栈是项目文档的"底层假设"，变更后所有基于旧假设的文档都会误导后续开发
**避坑**: 不要只改代码不改文档；不要只改顶层文档不改子目录文档；不要保留旧目录"以后可能用到"
**适用举例**: REST 迁移 GraphQL（API 文档全改）、MySQL 迁移 PostgreSQL（SQL 方言/ORM 配置）、Vue 2 升级 Vue 3（组件写法/构建配置）、单体拆微服务（目录结构/通信方式）

<details>
<summary>原始案例</summary>

**项目场景**: 旅游 Agent 项目，中途用户要求"前端用 Vue 框架，后端用 LangChain"。已生成的 16 个 CLAUDE.md 文件、19 步执行计划、前端目录结构全部基于旧技术栈（httpx + 纯 HTML/JS）
**具体做法**: 全量重写执行计划（19步→17步），更新 6 个过时 CLAUDE.md（collector 引用旧提取方式、prompts 引用旧函数签名、memory 列名不匹配、routers API 路径缺少前缀、output 数据结构不一致），删除 3 个旧目录（api/store/components），新建 4 个 CLAUDE.md（src/ 子目录），补充 .env.example 和 04-report
</details>

## 实现交付物检查清单

> 沉淀于 2026-04-13，来源：完成 17 步实现后用户发现遗漏了 .env.example、04-report、子目录 CLAUDE.md

**通用场景**: 多步骤实现任务完成后，遗漏了非代码交付物（配置模板、报告、文档、规范文件），直到用户或后续开发者发现缺失
**识别信号**: 实现任务包含 5+ 步骤；涉及新建目录；有 Skill 流水线（拆解→复用→计划→实现）；涉及 CLAUDE.md 文档体系
**通用做法**:
  1. 实现开始前，从执行计划提取**预期交付物清单**（代码文件 + 文档 + 配置）
  2. 每步完成后勾选对应交付物
  3. 全部步骤完成后，执行**交付物扫描**：对照计划文件清单逐项检查是否存在
  4. 必查项：配置模板(.env.example)、实现报告(04-report/)、每个新建目录的 CLAUDE.md
  5. 缺失项立即补充，不为"以后再补"
**原因**: 实现过程中注意力集中在代码逻辑上，非代码交付物容易被遗忘。但它们是后续维护和协作的基础
**避坑**: 不要把"实现完所有步骤"等同于"交付完成"。代码只是交付物的一部分
**适用举例**: SDK 开发（缺 README/CHANGELOG）、微服务搭建（缺 Docker 配置/健康检查）、CI/CD 配置（缺 .env 模板/部署文档）、任何多模块项目（缺子模块规范文档）

<details>
<summary>原始案例</summary>

**项目场景**: 旅游 Agent 完成 S01-S17 全部代码实现后，用户问"你是不是忘记生成什么东西了？"。遗漏了：.env.example 配置模板、04-report/ 实现报告、travel-web/src/ 子目录的 CLAUDE.md 文件
**具体做法**: 补充 .env.example（所有环境变量说明），生成 04-report/README.md（17 步完成详情、文件清单、启动说明），新建 4 个 src/ 子目录 CLAUDE.md
</details>

## 需求分析后模块设计对齐

> 沉淀于 2026-04-13，来源：旅游 Agent 需求拆解后直接跳到执行计划，未分析现有目录结构和模块设计是否匹配需求

**通用场景**: 需求拆解/分析完成后，直接进入执行计划或编码阶段，跳过了"对照现有架构确认模块划分、目录结构、接口设计是否合理"的关键步骤
**识别信号**: 项目已有代码/目录结构（非空项目）；需求涉及新模块或模块间交互；功能拆解产生了新目录/新文件
**通用做法**:
  1. 需求拆解完成后，**增加一步"模块设计对齐"**
  2. 检查现有目录结构，确认新功能应该放在哪里
  3. 确认每个新建目录的 CLAUDE.md 是否需要更新或创建
  4. 确认模块间依赖关系是否引入了循环依赖
  5. 将对齐结果写入拆解报告的补充段落
**原因**: 需求拆解从功能角度拆，但实现受限于已有代码结构。不对齐可能导致后续实现时频繁调整目录/模块
**避坑**: 不要假设"拆解完就能直接写代码"。已有项目有惯性，新功能需要适配现有结构
**适用举例**: 在已有微服务中加新服务（需确认服务边界）、在已有前端项目中加新页面（需确认路由/状态管理方案）、在已有类库中加新模块（需确认包结构和导出方式）

<details>
<summary>原始案例</summary>

**项目场景**: 旅游 Agent 项目已有 16 个 CLAUDE.md 文件和基本目录结构，用户提出需求变更（LangChain + Vue 3）。直接跳到重写计划，未分析已有 CLAUDE.md 哪些需要更新、哪些目录需要删除/新建
**具体做法**: 事后逐个检查 16 个 CLAUDE.md，发现 6 个过时（collector/prompts/memory/routers/output/intent），4 个缺失（src/ 子目录），3 个旧目录需删除
</details>

## 方案先行沟通

> 沉淀于 2026-04-13，来源：用户提出技术栈变更后，直接重写计划并开始实现，未先告知"我会这样做、分这几个模块完成"

**通用场景**: 收到需求变更或新需求后，直接进入实现，未先向用户展示方案、功能划分、影响范围。用户发现方向不对时已经做了大量工作
**识别信号**: 用户说"换成 X"/"我需要 Y"；需求涉及 3+ 文件或模块变更；技术方案有多种选择
**通用做法**:
  1. 收到需求后，**先输出方案概要**：我打算怎么做，分几个功能模块，每个模块做什么
  2. 明确列出**影响范围**：哪些文件/目录需要改、新增、删除
  3. 标注**关键决策点**：需要用户确认的选型或权衡
  4. 用户确认后再进入详细计划或实现
  5. 确认内容包括：功能划分是否OK、优先级是否正确、有没有遗漏
**原因**: 方案层面的理解偏差成本远低于实现层面。先沟通方案，避免做了一半发现方向不对
**避坑**: 不要用"全量更新计划"代替方案沟通——计划是细化的施工图，方案是粗略的蓝图。先确认蓝图再画施工图
**适用举例**: 任何涉及架构决策的需求（选型变更、功能扩展、性能优化）、用户需求模糊的场景（先对齐理解再动手）、多方案可选的场景（列出选项让用户选）

<details>
<summary>原始案例</summary>

**项目场景**: 用户说"前端用 Vue 框架，后端用 LangChain 快速开发"。我直接重写了整个 03-plan 执行计划（19步→17步），更新了 CLAUDE.md，然后开始逐步骤实现。用户确认后我也没有先解释"我打算这样做"就开始编码
**具体做法**: 应该先告诉用户"我打算将执行计划调整为 17 步：后端用 LangChain ChatOpenAI 替代 httpx（简化 3 个文件），前端用 Vue 3 + Vite 替代纯 HTML（重构目录结构），预计影响 X 个文件"，确认后再重写
</details>

## 交付物对照扫描（TODO 清单验证）

> 沉淀于 2026-04-13，来源：用户反馈"在输出文档的过程中需要根据任务基本生成的文档去完成一个 TODO 扫描"

**通用场景**: 流水线式工作流（拆解→复用→计划→实现）中，上游 Skill 生成的文档列出了预期交付物，但下游 Skill 执行后未对照上游清单验证是否全部完成
**识别信号**: 使用多 Skill 流水线（拆解→复用→计划→实现）；上游文档中列出了文件清单/模块清单/验收标准；实现完成后声称"全部完成"
**通用做法**:
  1. 实现开始前，从上游文档（拆解报告/执行计划）提取**预期交付物 TODO 列表**
  2. 每完成一步，在 TODO 列表中勾选对应项
  3. 全部步骤完成后，**扫描 TODO 列表**：逐项确认文件是否存在、内容是否与计划一致
  4. 发现遗漏立即补充，不留空项
  5. 在最终报告中附加"TODO 扫描结果"
**原因**: 多步骤流程中每步只关注自己的范围，容易遗漏跨步骤的交付物（如配置文件、文档、报告）
**避坑**: 不要依赖记忆——步骤多了必定遗漏。必须用显式清单对照
**适用举例**: 任何流水线式开发流程（设计→开发→测试→部署）、多步骤配置流程（环境→依赖→代码→验证）、文档生成流水线（大纲→初稿→校审→终稿）

<details>
<summary>原始案例</summary>

**项目场景**: 旅游 Agent 使用 task-breakdown → code-reuse-finder → impl-planner → code-implementer 流水线。impl-planner 生成的 README.md 列出了完整的文件结构总览和步骤清单，但 code-implementer 完成后未对照该清单扫描，遗漏了 .env.example、04-report、部分 CLAUDE.md
**具体做法**: code-implementer 完成所有步骤后，应读取 impl-planner 的 README.md 文件结构总览，逐项检查每个列出的文件是否存在，发现遗漏立即补充
</details>
