## 接口测试模式

### 第一步：识别接口

1. 读取项目中的路由定义文件（如 `routes/`、`controllers/`、`api/`、`router.*`、`app.py` 等）
2. 提取所有接口端点：

```
| 方法 | 路径 | 请求参数 | 响应格式 | 认证方式 |
|------|------|---------|---------|---------|
| POST | /api/auth/login | { email, password } | { token, user } | 无 |
| GET  | /api/user/:id | path: id | { id, name, email } | Bearer Token |
```

3. 如果用户指定了特定接口，只分析该接口；否则分析所有接口

### 第二步：生成接口测试

**每个接口至少覆盖**：

| 维度 | 说明 | 示例 |
|------|------|------|
| 正常请求 | 正确参数返回预期状态码 | POST /login → 200 |
| 参数缺失 | 缺少必填参数 | POST /login 无 password → 400 |
| 参数非法 | 类型错误、格式错误 | email 不是邮箱格式 → 422 |
| 认证缺失 | 未携带认证信息 | GET /user/1 无 Token → 401 |
| 权限不足 | 无权访问的资源 | 删除他人资源 → 403 |
| 边界情况 | 空值、超长字符串、特殊字符 | name 500字 → 取决于业务 |
| 重复操作 | 重复创建、并发请求 | POST 创建已存在的 → 409 |

**接口测试工具适配**：

| 语言/框架 | 推荐工具 | 示例 |
|----------|---------|------|
| JS/TS | supertest + vitest/jest | `request(app).post('/api/login').send({...})` |
| Python | pytest + httpx / requests | `client.post('/api/login', json={...})` |
| Java | Spring MockMvc / REST Assured | `mockMvc.perform(post('/api/login')...)` |
| Go | net/http/httptest | `httptest.NewServer(...)` |
| Rust | reqwest + actix-web test | |
| C# | WebApplicationFactory | `client.PostAsync('/api/login', ...)` |

### 第三步：运行接口测试

同单元测试第三步。运行完后**继续执行第五步：生成配套文件**。

## 批量测试模式

仅在用户明确要求测试全部项目时进入此模式。

### 第一步：发现子项目

扫描项目根目录，发现所有子项目及其语言/框架：

```
发现以下子项目：
├── frontend/      [JS/TS]    Vitest    12 个源文件
├── backend_agent/ [Python]   pytest    8 个源文件
├── backend_base/  [Java]     JUnit 5   15 个源文件
└── 共 35 个源文件需要生成测试
```

### 第二步：逐子项目处理

对每个子项目**独立处理**，互不干扰：
1. 读取该子项目的依赖文件，确定测试框架
2. 扫描该子项目内的源文件和已有测试
3. 按该语言的框架适配规则生成测试文件
4. 测试文件只生成在该子项目目录内
5. 每个子项目独立运行测试、独立报告
6. 每个子项目完成后**执行第五步：生成配套文件**（run-tests.sh、run-tests.ps1、README.md）

## 通用功能测试指导

> 基于 `.claude/doc/testing-standard/` 规范，适用于任何功能模块的测试设计

### 一、测试分层原则

```
        ╱  E2E 测试  ╲          ~10% — 关键用户路径/主链路
       ╱───────────────╲
      ╱   集成测试       ╲        ~20% — 模块间协作、外部依赖交互
     ╱───────────────────╲
    ╱     单元测试         ╲      ~70% — 函数/方法级别纯逻辑
   ╱───────────────────────╲
```

设计测试时先按分层规划，再逐层编写：
1. **先写单元测试**：覆盖所有纯函数、参数校验、状态转换、错误分支
2. **再写集成测试**：覆盖模块间调用链、编排顺序、降级路径
3. **最后写 E2E 测试**：只覆盖关键主链路，依赖真实环境

### 二、功能单元 → 测试用例映射规则

对每个功能单元，按以下四个维度生成用例：

| 维度 | 必须覆盖 | 典型场景 |
|------|---------|---------|
| 正常路径 | 是 | 合法输入 → 预期输出 |
| 边界值 | 是 | 空值、零值、极值、默认参数、空列表 |
| 异常路径 | 是 | 非法输入 → 预期错误、外部依赖失败 |
| 状态变体 | 按需 | 对象不同初始状态、并发调用、重复调用 |

**用例数量估算**：每个功能单元通常 3~6 条用例（正常×1 + 边界×1~2 + 异常×1~2 + 变体×0~1）

### 三、Mock 边界判定（通用决策树）

```
被测功能是否依赖外部系统？（网络/文件/数据库/子进程/时间）
├── 是 → 必须 Mock
│   ├── 网络（HTTP/WebSocket/RPC）     → vi.mock('http') / vi.mock('ws') / vi.mock('fetch')
│   ├── 文件系统                        → vi.mock('fs') 或使用临时目录
│   ├── 数据库                          → vi.mock 驱动 或 内存数据库
│   ├── 子进程                          → vi.mock('child_process')
│   └── 时间                            → vi.useFakeTimers()
│
└── 否 → 使用真实实现
    ├── 真实实现太慢？（>100ms）
    │   ├── 是 → Mock
    │   └── 否 → 真实实现
    └── 真实实现有副作用？（修改全局状态）
        ├── 是 → Mock 或 隔离
        └── 否 → 真实实现
```

**核心规则**：
- 只 Mock 外部边界，不 Mock 被测模块内部函数
- 不 Mock 值对象（DTO/VO/类型）
- 优先验证输出状态，而非验证调用次数

### 四、编排/流程类功能的测试模式

当被测模块是一个编排器（调用多个子模块完成一个流程）时：

```
编排器
  ├── 子模块 A（已独立测试）
  ├── 子模块 B（已独立测试）
  └── 子模块 C（已独立测试）
```

**测试重点**：
1. **调用顺序**：验证子模块被按正确顺序调用
2. **数据传递**：验证上游输出正确传递给下游
3. **降级路径**：某个子步骤失败时，验证降级行为（如优雅关闭超时 → 强制终止）
4. **状态一致性**：编排完成后，验证整体状态一致（如进程已终止 且 连接已断开 且 资源已清理）

**Mock 策略**：Mock 所有子模块，只验证编排逻辑，不重复测试子模块内部。

### 五、异步操作测试要点

| 场景 | 测试方法 | 注意事项 |
|------|---------|---------|
| Promise/async | `await` + `expect` | 必须处理 rejection |
| 轮询重试 | `vi.useFakeTimers()` + `vi.advanceTimersByTime()` | 验证重试次数和间隔 |
| 事件监听 | Mock emitter，手动触发事件 | 验证回调参数正确 |
| 超时 | `vi.useFakeTimers()` 推进时间 | 验证超时后抛出错误 |
| 并发操作 | `Promise.all` + 多次 resolve | 验证无竞态条件 |

### 六、覆盖率目标

| 模块类型 | 语句覆盖率 | 分支覆盖率 |
|---------|-----------|-----------|
| 核心模块（支付/认证/协议层） | ≥ 80% | ≥ 70% |
| 普通业务模块 | ≥ 70% | ≥ 60% |
| 编排/门面模块 | ≥ 70% | ≥ 60% |
| 新增/修改代码 | ≥ 75% | ≥ 65% |

### 七、测试反模式（必须避免）

| 反模式 | 正确做法 |
|--------|---------|
| 测试实现细节（私有方法、内部变量名） | 测试公共行为和接口 |
| 过度 Mock（被测模块内部也被 mock） | 只 mock 外部边界 |
| 测试间共享可变状态 | 每个测试独立准备数据 |
| 硬编码 sleep 等待 | 轮询/事件驱动/fake timer |
| 只测 Happy Path | 边界 + 异常全覆盖 |
| 截图对比测试（样式微小变化就失败） | 只对关键视觉元素做 snapshot |
