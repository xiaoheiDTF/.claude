# Java Code Review 规则

> 适用于所有架构模式（DDD / MVC）
> 最后更新：2026-04-11

---

## 一、Review 核心原则

> 只要代码确实改善了系统健康度，就应该批准通过，即使它不完美。
> —— [Google Engineering Practices](https://google.github.io/eng-practices/review/reviewer/standard.html)

- 没有"完美代码"，只有"更好的代码"
- 追求**持续改进**，而非一次到位
- 技术事实和数据优先于个人偏好
- 不影响代码健康度的建议，加 **"Nit:"** 前缀表示可选
- 不要让 PR 因为意见分歧而无限期挂起

## 二、Review Checklist（审查清单）

### P0 — 必须检查（阻塞合并）

#### 2.1 功能正确性
- [ ] 代码是否正确实现了需求
- [ ] 边界条件是否处理
- [ ] 并发场景是否安全（共享资源、锁、线程安全集合）
- [ ] 空指针风险是否消除（Optional、null check、@NonNull）

#### 2.2 安全性
- [ ] 用户输入是否验证和清理
- [ ] SQL 是否参数化（无 SQL 注入风险）
- [ ] 是否有 XSS 风险
- [ ] 敏感信息是否出现在日志或响应中
- [ ] 密码/密钥是否正确加密存储

#### 2.3 分层依赖
- [ ] 是否有反向依赖（如 Service 调用 Controller）
- [ ] DO/PO 是否越层传递（不应出现在 Controller 层）
- [ ] 分层职责是否清晰（如 Controller 是否包含业务逻辑）

### P1 — 重要检查（建议修复）

#### 2.4 设计合理性
- [ ] 类设计是否遵循单一职责
- [ ] 接口设计是否合理（不为每个实现都创建接口）
- [ ] 异常处理是否正确（catch 顺序、异常传播）
- [ ] 是否有不必要的复杂度（嵌套 if、长 switch）

#### 2.5 代码质量
- [ ] 命名是否清晰有意义
- [ ] 方法长度是否合理（≤ 80 行）
- [ ] 是否有重复代码（DRY）
- [ ] 是否有魔法值
- [ ] 集合操作是否安全（foreach 中无 remove）

#### 2.6 性能
- [ ] 循环中是否有冗余计算
- [ ] 数据库查询是否优化（N+1、索引）
- [ ] 资源（流、连接）是否正确关闭
- [ ] 是否有潜在的内存泄漏

### P2 — 建议检查（不阻塞合并）

- [ ] 代码格式是否一致
- [ ] 注释是否清晰
- [ ] Javadoc 是否完整
- [ ] 测试覆盖率是否达标
- [ ] 是否使用了更好的设计模式

## 三、Review 流程规则

### 3.1 提交者（Author）

- 【强制】PR 描述包含：做了什么、为什么做、如何测试
- 【强制】一个 PR 只做一件事，建议 < 400 行
- 【推荐】自审一遍再提交

### 3.2 审查者（Reviewer）

- 【强制】24 小时内完成 Review
- 【推荐】反馈要具体可操作，不只说"不好"
- 【推荐】区分 "必须改" 和 "建议改"，后者加 "Nit:"
- 【推荐】以学习心态 Review

### 3.3 冲突解决

1. PR 评论中基于文档达成共识
2. 评论无法解决 → 视频会议沟通
3. 仍无法解决 → 升级 Tech Lead
4. **绝对不要让 PR 无限挂起**

## 四、常见 Review 陷阱

| 陷阱 | 规避方法 |
|------|---------|
| 走马观花 | 专门留时间，保持 PR 小 |
| 只盯语法 | 优先关注功能和设计，格式交给工具 |
| 吹毛求疵 | 区分 "Nit:" 和必须修改项 |
| 无建设性反馈 | 提供具体改进建议 |
| 无视规范 | 用 Checkstyle/PMD 自动化 |
