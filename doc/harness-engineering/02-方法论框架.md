# 02 — 方法论框架

> **定位**: Harness Engineering 的实践方法和技术体系  
> **来源**: OpenAI 官方博客 + 社区实践总结

---

## 三大支柱

Harness Engineering 的核心方法论建立在三大支柱之上：

```
环境设计（Environment Design）
    ↓
反馈循环（Feedback Loops）
    ↓
约束执行（Constraint Enforcement）
```

这三者形成一个闭环：环境定义了 Agent 能做什么，反馈循环告诉 Agent 做得对不对，约束执行机械化地阻止错误。

---

## 支柱一：环境设计（Environment Design）

### 核心原则：仓库即操作系统

**所有架构决策、约束规则、设计文档都必须编码到 Git 仓库中，成为版本化、可执行的文件。**

#### 标准仓库结构

```
project/
├── AGENTS.md                    # 导航地图（~100行）
├── docs/
│   ├── architecture/
│   │   ├── README.md           # 架构概览
│   │   ├── layers.md           # 分层架构和依赖规则
│   │   └── boundaries.md       # 模块边界定义
│   ├── design/
│   │   ├── feature-auth.md     # 认证功能设计
│   │   └── feature-api.md      # API 设计
│   └── conventions/
│       ├── README.md           # 代码规范总览
│       ├── naming.md           # 命名规范
│       └── testing.md          # 测试规范
├── scripts/
│   ├── lint-deps.sh            # 依赖方向检查
│   ├── lint-quality.sh         # 质量规则检查
│   └── validate.sh             # 统一验证管道
├── src/                        # 应用代码
└── tests/                      # 测试代码
```

#### AGENTS.md 的设计原则

**AGENTS.md 是地图，不是手册。**

```markdown
# AGENTS.md

## 项目概览
这是一个 Web 应用，采用三层架构。详见 docs/architecture/README.md

## 架构约束
- 依赖方向：UI → Service → Repository
- 详见 docs/architecture/layers.md

## 开发流程
1. 阅读相关设计文档（docs/design/）
2. 实现功能
3. 运行 `scripts/validate.sh` 验证
4. 提交 PR

## 代码规范
详见 docs/conventions/README.md

## 常见任务
- 添加新 API：参考 docs/design/feature-api.md
- 添加新测试：参考 docs/conventions/testing.md
```

**关键点**：
- 总长度控制在 100 行左右
- 只提供导航，不重复内容
- 指向具体文档，而不是内联所有信息

### 上下文管理策略

#### 策略一：渐进式披露（Progressive Disclosure）

不要一次性把所有信息塞给 Agent。设计信息的层级结构：

```
第一层（必读）: AGENTS.md
    ↓
第二层（按需）: docs/architecture/README.md
    ↓
第三层（具体）: docs/design/feature-X.md
```

**实践建议**：
- 第一层：100 行以内，每次任务都读
- 第二层：500 行以内，处理相关模块时读
- 第三层：不限长度，处理具体功能时读

#### 策略二：按需加载（On-demand Loading）

让 Agent 自己决定需要读什么：

```markdown
## 认证功能

如果你需要修改认证逻辑，请先阅读：
- docs/design/feature-auth.md
- docs/architecture/security.md
```

#### 策略三：文档分层（Map > Manual）

| 文档类型 | 长度 | 更新频率 | 读取频率 |
|---------|------|---------|---------|
| **地图**（AGENTS.md） | ~100 行 | 低 | 每次任务 |
| **架构**（architecture/） | ~500 行/文件 | 中 | 按模块 |
| **设计**（design/） | 不限 | 高 | 按功能 |
| **规范**（conventions/） | ~300 行/文件 | 低 | 按需 |

## 会话协议（Session Protocol）

> **来源**: Anthropic "Effective Harnesses for Long-Running Agents" + 社区最佳实践总结

在 Harness Engineering 中，**会话（Session）**不是随意的对话，而是有严格生命周期的执行单元。一个设计良好的会话协议可以显著降低跨会话的累积错误。

### 一致的会话生命周期

每个会话应遵循可预测的顺序：

```
Orient（定向）     → 读取进度笔记、任务列表、最近 git 历史
    ↓
Setup（准备）      → 运行 init 脚本启动开发服务器/前置条件
    ↓
Verify（验证基线） → 测试现有功能是否仍然工作（防止上一会话引入 Bug）
    ↓
Select（选择任务） → 选择最高优先级的未完成项
    ↓
Implement（实现）  → 构建功能
    ↓
Test（测试）       → 通过实际 UI/API 验证，不只是单元测试
    ↓
Update（更新状态） → 标记任务完成、提交、写进度笔记
    ↓
Clean Exit（干净退出）→ 确认应用处于可用状态
```

**关键原则**：
- **一个会话一个任务** — 防止上下文耗尽，保持会话聚焦和可恢复
- **先验证基线再构建** — 跨会话的累积 Bug 是最常见的失败模式之一
- **上下文重置优于压缩** — 每次会话创建新客户端/上下文，而非仅依赖压缩；结构化交接文件（handoff artifacts）桥接会话间的 gap

### 状态持久化策略

Agent 的上下文窗口是短暂的，所有必须跨会话存活的状态都应写入磁盘：

| 状态类型 | 格式 | 规则 |
|---------|------|------|
| **任务/功能列表** | JSON（而非 Markdown） | 永不删除或重排项目，只翻转状态从 incomplete 到 complete |
| **进度笔记** | 自由文本 | 每会话结束时写入：完成了什么、发现的 Bug、下一步做什么 |
| **计划/规格文件** | Markdown | 放在项目目录中供 Agent 参考 |
| **Git 历史** | 描述性提交 | 作为恢复机制，Agent 在会话开始时读取最近提交 |

## 支柱二：反馈循环（Feedback Loops）

### 核心原则：快速失败，自动修正

**Agent 应该在几秒内知道自己做错了什么，而不是等到人类 Code Review。**

### 验证机制的层级

#### 第一层：静态检查（秒级反馈）

```bash
# scripts/lint-deps.sh
# 检查依赖方向是否违反架构约束

#!/bin/bash
set -e

echo "Checking dependency directions..."

# UI 层不应该依赖 Repository 层
if grep -r "import.*Repository" src/ui/; then
    echo "ERROR: UI layer should not import Repository layer"
    exit 1
fi

# Service 层不应该依赖 UI 层
if grep -r "import.*UI" src/service/; then
    echo "ERROR: Service layer should not import UI layer"
    exit 1
fi

echo "✓ Dependency directions are correct"
```

```bash
# scripts/lint-quality.sh
# 检查代码质量规则

#!/bin/bash
set -e

echo "Checking code quality rules..."

# 单文件不超过 500 行
find src/ -name "*.ts" | while read file; do
    lines=$(wc -l < "$file")
    if [ "$lines" -gt 500 ]; then
        echo "ERROR: $file has $lines lines (max 500)"
        exit 1
    fi
done

# 禁止使用 console.log
if grep -r "console.log" src/; then
    echo "ERROR: Use structured logging instead of console.log"
    exit 1
fi

echo "✓ Code quality rules passed"
```

#### 第二层：单元测试（分钟级反馈）

```bash
# 运行单元测试
npm test
```

**关键点**：
- 测试应该快（< 5 分钟）
- 测试应该稳定（无 flaky tests）
- 测试失败信息应该清晰

#### 第三层：集成测试（10 分钟级反馈）

```bash
# 运行集成测试
npm run test:integration
```

#### 第四层：端到端验证（小时级反馈）

```bash
# scripts/validate.sh
# 完整验证管道

#!/bin/bash
set -e

echo "Running full validation pipeline..."

./scripts/lint-deps.sh
./scripts/lint-quality.sh
npm run lint
npm test
npm run test:integration
npm run build

echo "✓ All validations passed"
```

### Agent 互审机制

**让 Agent 互相 Review 代码，而不是等人类。**

OpenAI 的实践：
1. Agent A 提交 PR
2. Agent B 自动触发，执行 Code Review
3. Agent B 提出修改建议或直接修改
4. 人类只需要做最终的"批准/拒绝"判断

**实现方式**：
- GitHub Actions + AI Agent
- 自动触发 Review 流程
- 检查架构一致性、代码规范、测试覆盖率

#### 生成与评估分离（Generator / Evaluator 模式）

> **来源**: Anthropic 最佳实践总结

Agent  consistently 对自己的工作评价过高。**将生成与评估分离是构建诚实反馈循环的关键。**

| 角色 | 职责 | 设计要点 |
|------|------|---------|
| **Generator（生成器）** | 根据规格实现功能 | 专注实现，不做自我评判 |
| **Evaluator（评估器）** | 以用户方式测试应用，对输出打分 | 被调优成"怀疑论者"，远比让生成器自我批评更容易 |

**Sprint Contract（冲刺契约）**：

在复杂的实现块之前，让生成器和评估器协商"完成"的定义：
1. 生成器提议：它将构建什么、如何验证成功
2. 评估器审查该提议
3. 双方达成一致后才开始实现

这弥合了高层规格和可测试实现之间的鸿沟。

### 持续垃圾回收（Continuous Garbage Collection）

**不要等到"技术债务爆炸"才重构，而是持续小额还款。**

```bash
# 定期扫描架构偏差
# 自动开 PR 修正

#!/bin/bash

# 扫描过长的文件
find src/ -name "*.ts" | while read file; do
    lines=$(wc -l < "$file")
    if [ "$lines" -gt 400 ]; then
        echo "File $file is getting large ($lines lines)"
        # 触发 Agent 任务：重构这个文件
    fi
done

# 扫描重复代码
# 触发 Agent 任务：提取公共逻辑

# 扫描过时的依赖
# 触发 Agent 任务：升级依赖
```

**关键点**：
- 自动化触发，不依赖人类记忆
- 小步快跑，每次只改一点
- 高频执行（每天/每周）

### "成功应静默，失败才发声"原则

> **来源**: HumanLayer 团队实践

HumanLayer 团队早期犯了一个看似合理的错误：每次 Agent 修改代码后都运行完整测试套件。结果 **4000 行通过的测试输出涌入上下文窗口**，Agent 开始对刚读到的测试文件产生幻觉，丢失了对实际任务的追踪。

他们总结出一条反直觉的设计原则：

> **"成功应该是沉默的，只有失败才应该发出声音。"**

**实现方式**：

```bash
# Hook 脚本示例
# 当 Agent 停止工作时自动运行检查

npm run format
npm run type-check

# 如果一切通过 → 完全静默，不向上下文注入任何内容
# 如果失败 → 只输出错误信息，并用退出码告诉 Harness 重新激活 Agent
```

**设计智慧**：当验证通过时完全静默不污染上下文，失败时才输出错误信息并触发 Harness 重新激活 Agent。这解决了早期实践的痛点——大量通过测试的输出淹没了上下文窗口，导致 Agent 对最近读取的测试文件产生幻觉。

## 支柱三：约束执行（Constraint Enforcement）

### 核心原则：约束 > 指导

**不要告诉 AI "应该怎么做"，而是机械化地验证"做得对不对"。**

### 约束的三个层次

#### 层次一：不可能违反（编译时）

通过类型系统、编译器、Linter 强制执行。

```typescript
// 使用 TypeScript 的类型系统强制依赖方向
// UI 层只能看到 Service 接口，看不到 Repository

// src/service/UserService.ts
export interface UserService {
    getUser(id: string): Promise<User>;
}

// src/ui/UserPage.ts
import { UserService } from '../service/UserService';
// ✓ 允许

// import { UserRepository } from '../repository/UserRepository';
// ✗ 编译错误：UI 层不应该导入 Repository
```

#### 层次二：很难违反（CI 时）

通过自动化脚本在 CI 中检查。

```yaml
# .github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Check dependency directions
        run: ./scripts/lint-deps.sh
      - name: Check code quality
        run: ./scripts/lint-quality.sh
      - name: Run tests
        run: npm test
```

**关键点**：
- CI 失败 = PR 无法合并
- 反馈快（< 5 分钟）
- 错误信息清晰

#### 层次三：容易发现（运行时）

通过监控、日志、告警在运行时发现问题。

```typescript
// 运行时断言
function processOrder(order: Order) {
    assert(order.items.length > 0, "Order must have items");
    assert(order.total > 0, "Order total must be positive");
    // ...
}
```

### 约束的可执行性

| 约束类型 | 传统做法（指导） | Harness 做法（约束） |
|---------|---------------|------------------|
| **架构分层** | "请遵循 Clean Architecture" | `lint-deps.sh` 检查依赖方向 |
| **文件大小** | "单文件不要超过 500 行" | `lint-quality.sh` 自动拒绝 |
| **日志规范** | "使用结构化日志" | `grep -r "console.log"` 返回非零则 CI 失败 |
| **命名规范** | "使用 camelCase" | ESLint 规则自动检查 |
| **测试覆盖率** | "保持 80% 以上" | CI 中检查覆盖率，低于阈值则失败 |

### 约束的演进

**约束不是一次性设计的，而是随着项目演进逐步添加的。**

```
第一周：基本的 Linter + 测试
    ↓
第二周：添加依赖方向检查
    ↓
第四周：添加文件大小检查
    ↓
第八周：添加架构边界测试
    ↓
持续：根据实际问题添加新约束
```

**核心原则**：

> **"每当发现 Agent 犯错，就花时间设计一个解决方案，确保 Agent 永远不会再犯同样的错误。"**

这不是在调教 Agent，而是在升级环境。当一个错误被环境约束解决掉之后，**所有未来经过这个环境的 Agent 都会自动受益**——不需要逐个修复 Prompt，不需要逐个更新指令。

### 子 Agent 策略（Subagent Strategy）

> **来源**: Anthropic 长时运行 Agent 最佳实践

**主上下文窗口应作为调度器运行**，将昂贵的工作卸载给子 Agent：

| 操作类型 | 策略 | 原因 |
|---------|------|------|
| **只读操作**（搜索、分析） | 高并行度展开子 Agent | 不修改文件，无冲突风险 |
| **写操作**（构建、测试） | 限制并行度 | 避免反向压力和冲突 |
| **结果汇总** | 子 Agent 先总结，再传入主上下文 | 避免原始输出淹没主上下文 |

**关键原则**：
- 确定性加载相同的核心文件每个循环 —— 计划和规格应每次迭代都加载，确保 Agent 始终有相同的基础
- 不要让 Agent 假设代码不存在 —— 代码搜索（ripgrep）是非确定性的，指示 Agent 在实现前先搜索，避免重复实现

## 完整工作流示例

### 场景：添加新功能

```
1. 人类：定义意图
   "添加用户注册功能"

2. Agent：读取上下文
   - 读取 AGENTS.md
   - 读取 docs/architecture/README.md
   - 读取 docs/design/feature-auth.md

3. Agent：实现功能
   - 创建 src/service/RegistrationService.ts
   - 创建 src/ui/RegistrationPage.tsx
   - 创建 tests/RegistrationService.test.ts

4. Agent：自我验证
   - 运行 scripts/validate.sh
   - 所有检查通过

5. Agent：提交 PR
   - 自动生成 PR 描述
   - 触发 CI

6. Agent B：Code Review
   - 检查架构一致性
   - 检查测试覆盖率
   - 提出修改建议

7. 人类：最终判断
   - 批准或拒绝
   - 提供高层反馈
```

### 场景：修复 Bug

```
1. 人类：报告问题
   "用户注册时邮箱验证失败"

2. Agent：诊断
   - 读取相关代码
   - 读取相关测试
   - 读取日志

3. Agent：修复
   - 修改 src/service/RegistrationService.ts
   - 添加测试用例

4. Agent：验证
   - 运行 scripts/validate.sh
   - 确认 Bug 修复

5. Agent：提交 PR
   - 自动生成 PR 描述
   - 关联 Issue

6. 人类：确认
   - 验证修复效果
   - 合并 PR
```

## 关键实践总结

### 环境设计的关键

1. **所有知识进仓库** — Wiki、Slack、脑中的知识对 Agent 不存在
2. **地图 > 手册** — AGENTS.md 控制在 100 行
3. **渐进式披露** — 分层组织信息
4. **Agent 可读性优先** — 选择 Agent 能理解的技术栈

### 反馈循环的关键

1. **快速失败** — 秒级/分钟级反馈
2. **自动修正** — Agent 互审，持续垃圾回收
3. **分层验证** — 静态检查 → 单元测试 → 集成测试 → E2E
4. **清晰错误信息** — Agent 能理解如何修复

### 约束执行的关键

1. **约束 > 指导** — 机械化验证，不依赖 Agent "记住"
2. **不可能违反 > 很难违反 > 容易发现** — 分层设计约束
3. **可执行性** — 每个约束都有对应的自动化检查
4. **演进性** — 约束随项目成长逐步添加

## 常见陷阱

### 陷阱一：AGENTS.md 写成 1000 行手册

**错误做法**：
```markdown
# AGENTS.md (1000+ 行)

## 架构
我们使用三层架构...（500 行详细说明）

## 代码规范
命名规范：...（200 行详细说明）
测试规范：...（300 行详细说明）
```

**正确做法**：
```markdown
# AGENTS.md (~100 行)

## 架构
详见 docs/architecture/README.md

## 代码规范
详见 docs/conventions/README.md
```

### 陷阱二：约束规则不可执行

**错误做法**：
```markdown
# 代码规范

- 请遵循 Clean Architecture
- 单文件不要超过 500 行
- 使用结构化日志
```

**正确做法**：
```bash
# scripts/lint-deps.sh
# 检查依赖方向

# scripts/lint-quality.sh
# 检查文件大小、日志使用
```

### 陷阱三：过早优化

**错误做法**：
- 项目第一天就设计完整的 Harness
- 添加大量"可能需要"的约束

**正确做法**：
- 从最小可行 Harness 开始
- 根据实际问题逐步添加约束
- 让 Harness 随项目成长

### 陷阱四：忽视人类判断

**错误做法**：
- 完全依赖 Agent，人类不做任何 Review
- 认为 Harness 能解决所有问题

**正确做法**：
- 人类负责高层决策、架构方向、产品判断
- Agent 负责执行、验证、重复劳动
- Harness 是工具，不是替代品
