# 06 — 实施指南

> **定位**: 如何在实际项目中应用 Harness Engineering  
> **来源**: OpenAI 实践 + 社区经验总结

---

## 前置条件评估

在开始之前，先评估你的项目是否适合 Harness Engineering。

### 评估清单

| 问题 | 理想答案 | 说明 |
|------|---------|------|
| **团队是否认可 AI 写代码？** | 是 | 如果团队抗拒，强推会失败 |
| **项目是新项目还是遗留系统？** | 新项目 | 新项目更容易，遗留系统需要先梳理架构 |
| **架构是否清晰稳定？** | 是 | 架构不稳定时，Harness 会频繁变化 |
| **是否有时间投入设计 Harness？** | 是 | 需要 1-2 周的前期投入 |
| **项目周期是否足够长？** | > 3 个月 | 短期项目无法回收 Harness 投入 |
| **是否需要大量重复代码？** | 是 | Agent 擅长重复性工作 |

### 适用性判断

```
✅ 高度适合：
- 新项目，从零开始
- 架构清晰（如三层架构、微服务）
- 需要快速交付
- 团队规模小（3-7 人）
- 大量 CRUD、配置、测试代码

⚠️ 需要评估：
- 遗留系统（需要先梳理架构）
- 架构探索期（等架构稳定后再上 Harness）
- 中等复杂度项目

❌ 不适合：
- 短期项目（< 1 个月）
- 创新算法研究
- 极致性能优化
- 团队完全不认可 AI
```

## 阶段一：最小可行 Harness（1-2 周）

### 目标

建立最基础的 Harness，让 Agent 能完成第一个任务。

### 步骤 1.1：创建 AGENTS.md（1 天）

**目标**：100 行左右的导航地图。

```markdown
# AGENTS.md

## 项目概览
这是一个 [项目类型]，采用 [架构模式]。

## 仓库结构
- `src/` - 应用代码
- `tests/` - 测试代码
- `docs/` - 文档
- `scripts/` - 工具脚本

## 架构
详见 docs/architecture/README.md

## 开发流程
1. 阅读相关设计文档
2. 实现功能
3. 运行 `scripts/validate.sh` 验证
4. 提交 PR

## 代码规范
详见 docs/conventions/README.md

## 常见任务
- 添加新功能：参考 docs/design/
- 修复 Bug：参考 docs/troubleshooting.md
```

**关键点**：
- 控制在 100 行以内
- 只提供导航，不重复内容
- 指向具体文档

### 步骤 1.2：创建架构文档（2 天）

**目标**：让 Agent 理解项目架构。

```markdown
# docs/architecture/README.md

## 架构概览
本项目采用三层架构：

```
UI 层（用户界面）
    ↓
Service 层（业务逻辑）
    ↓
Repository 层（数据访问）
```

## 依赖规则
- UI 层可以依赖 Service 层
- Service 层可以依赖 Repository 层
- 禁止反向依赖
- 禁止跨层依赖

详见 docs/architecture/layers.md

## 模块划分
- `auth` - 认证授权
- `user` - 用户管理
- `order` - 订单管理

详见 docs/architecture/modules.md
```

**关键点**：
- 清晰的架构图
- 明确的依赖规则
- 模块边界定义

### 步骤 1.3：创建约束脚本（2 天）

**目标**：机械化验证架构约束。

```bash
#!/bin/bash
# scripts/lint-deps.sh
# 检查依赖方向

set -e

echo "Checking dependency directions..."

# UI 层不应该依赖 Repository 层
if grep -r "import.*Repository" src/ui/; then
    echo "ERROR: UI layer should not import Repository layer"
    exit 1
fi

# Service 层不应该依赖 UI 层
if grep -r "import.*UI" src/service/; then
    echo "ERROR: Service layer should not import UI layer"
    exit 1
fi

echo "✓ Dependency directions are correct"
```

```bash
#!/bin/bash
# scripts/lint-quality.sh
# 检查代码质量规则

set -e

echo "Checking code quality rules..."

# 单文件不超过 500 行
find src/ -name "*.ts" | while read file; do
    lines=$(wc -l < "$file")
    if [ "$lines" -gt 500 ]; then
        echo "ERROR: $file has $lines lines (max 500)"
        exit 1
    fi
done

# 禁止使用 console.log
if grep -r "console.log" src/; then
    echo "ERROR: Use structured logging instead of console.log"
    exit 1
fi

echo "✓ Code quality rules passed"
```

```bash
#!/bin/bash
# scripts/validate.sh
# 统一验证管道

set -e

echo "Running full validation pipeline..."

./scripts/lint-deps.sh
./scripts/lint-quality.sh
npm run lint
npm test

echo "✓ All validations passed"
```

**关键点**：
- 快速执行（< 5 分钟）
- 清晰的错误信息
- 可以在 CI 中运行

### 步骤 1.4：第一个 AI 生成的 PR（1 天）

**目标**：验证 Harness 可用。

```
任务：让 Agent 实现一个简单功能（如 Hello World API）

流程：
1. Agent 读取 AGENTS.md
2. Agent 读取 docs/architecture/README.md
3. Agent 实现功能
4. Agent 运行 scripts/validate.sh
5. Agent 提交 PR
```

**验证点**：
- Agent 能找到正确的文件位置
- Agent 遵循了架构约束
- 验证脚本通过
- PR 可以合并

### 步骤 1.5：CI/CD 集成（2 天）

**目标**：自动化验证。

```yaml
# .github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: Setup Node.js
        uses: actions/setup-node@v2
        with:
          node-version: '18'
      
      - name: Install dependencies
        run: npm install
      
      - name: Check dependency directions
        run: ./scripts/lint-deps.sh
      
      - name: Check code quality
        run: ./scripts/lint-quality.sh
      
      - name: Run linter
        run: npm run lint
      
      - name: Run tests
        run: npm test
```

**关键点**：
- 每次 PR 自动触发
- 失败则无法合并
- 反馈快（< 5 分钟）

## 阶段二：反馈循环建立（2-4 周）

### 目标

建立完整的反馈循环，让 Agent 能自我验证和互审。

### 步骤 2.1：Agent 互审机制（1 周）

**目标**：让 Agent 自动 Review PR。

```yaml
# .github/workflows/agent-review.yml
name: Agent Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  agent-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: Run Agent Review
        run: |
          # 调用 AI Agent API
          # 检查架构一致性
          # 检查测试覆盖率
          # 提出修改建议
          
          # 示例（需要根据实际 Agent 工具调整）
          curl -X POST https://your-agent-api.com/review \
            -H "Content-Type: application/json" \
            -d '{
              "pr_number": "${{ github.event.pull_request.number }}",
              "repo": "${{ github.repository }}"
            }'
```

**关键点**：
- 自动触发
- 检查架构一致性
- 检查测试覆盖率
- 提供清晰的反馈

### 步骤 2.2：自动化测试（1 周）

**目标**：完整的测试套件。

```typescript
// tests/architecture.test.ts
// 架构边界测试

import { describe, it, expect } from 'vitest';
import { analyzeImports } from './utils/import-analyzer';

describe('Architecture Boundaries', () => {
  it('UI layer should not import Repository layer', () => {
    const imports = analyzeImports('src/ui');
    const hasRepositoryImport = imports.some(imp => 
      imp.includes('Repository')
    );
    expect(hasRepositoryImport).toBe(false);
  });

  it('Service layer should not import UI layer', () => {
    const imports = analyzeImports('src/service');
    const hasUIImport = imports.some(imp => 
      imp.includes('UI')
    );
    expect(hasUIImport).toBe(false);
  });
});
```

**关键点**：
- 单元测试（快速）
- 集成测试（中速）
- 架构测试（验证边界）
- E2E 测试（慢速，可选）

### 步骤 2.3：质量门禁（1 周）

**目标**：设置合并标准。

```yaml
# .github/workflows/quality-gate.yml
name: Quality Gate

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: Check test coverage
        run: |
          npm run test:coverage
          # 检查覆盖率是否 >= 80%
          coverage=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
          if (( $(echo "$coverage < 80" | bc -l) )); then
            echo "ERROR: Test coverage is $coverage% (min 80%)"
            exit 1
          fi
      
      - name: Check code complexity
        run: |
          npm run complexity
          # 检查圈复杂度
      
      - name: Check bundle size
        run: |
          npm run build
          # 检查打包大小
```

**关键点**：
- 测试覆盖率要求
- 代码复杂度限制
- 打包大小限制
- 性能基准

## 阶段三：持续优化（持续）

### 目标

持续改进 Harness，适应项目演进。

### 实践 3.1：垃圾回收机制

**目标**：自动检测和修正架构偏差。

```bash
#!/bin/bash
# scripts/scan-drift.sh
# 扫描架构偏差

echo "Scanning for architecture drift..."

# 扫描过长的文件
echo "Checking file sizes..."
find src/ -name "*.ts" | while read file; do
    lines=$(wc -l < "$file")
    if [ "$lines" -gt 400 ]; then
        echo "WARNING: $file is getting large ($lines lines)"
        # 可以自动创建 Issue 或触发 Agent 任务
    fi
done

# 扫描重复代码
echo "Checking for code duplication..."
npx jscpd src/ --threshold 10

# 扫描过时的依赖
echo "Checking for outdated dependencies..."
npm outdated

echo "Drift scan complete"
```

**定期执行**：
- 每天/每周自动运行
- 发现问题自动开 Issue
- Agent 自动修正小问题

### 实践 3.2：文档更新流程

**目标**：保持文档与代码同步。

```markdown
# docs/MAINTENANCE.md

## 文档更新规则

### 何时更新文档？
- 架构变化 → 更新 docs/architecture/
- 新增模块 → 更新 docs/architecture/modules.md
- 新增约束 → 更新 docs/conventions/
- 新增功能 → 更新 docs/design/

### 如何验证文档？
- 运行 `scripts/validate-docs.sh`
- 检查文档链接是否有效
- 检查代码示例是否可运行
```

**自动化**：
- PR 中检查文档是否更新
- 定期扫描过时的文档
- Agent 自动更新文档

### 实践 3.3：约束规则演进

**目标**：根据实际问题添加新约束。

```
发现问题
    ↓
分析根因
    ↓
设计约束
    ↓
实现脚本
    ↓
添加到 CI
    ↓
更新文档
```

**示例**：

```
问题：发现多个文件使用了不同的日期格式化方式
    ↓
根因：没有统一的日期处理规范
    ↓
约束：强制使用 date-fns 库
    ↓
脚本：检查是否使用了其他日期库
    ↓
CI：添加到 lint-quality.sh
    ↓
文档：更新 docs/conventions/date-handling.md
```

### 实践 3.4：团队培训

**目标**：让团队理解 Harness Engineering。

**培训内容**：
1. Harness Engineering 基本概念（1 小时）
2. 如何与 Agent 协作（2 小时）
3. 如何设计约束（2 小时）
4. 如何维护 Harness（2 小时）

**实践练习**：
- 让 Agent 完成一个小功能
- Review Agent 的 PR
- 添加一个新的约束规则
- 修复一个架构偏差

## 常见陷阱及应对

### 陷阱一：AGENTS.md 写成 1000 行手册

**症状**：
- AGENTS.md 超过 500 行
- 包含大量详细说明
- Agent 每次都要读很长时间

**应对**：
- 严格控制在 100 行
- 只提供导航，不重复内容
- 详细内容放到单独文档

### 陷阱二：约束规则不可执行

**症状**：
- 文档说"请遵循 XX 规范"
- 但没有自动化检查
- Agent 经常违反规则

**应对**：
- 每个约束都要有对应的脚本
- 在 CI 中自动执行
- 违反约束则 PR 无法合并

### 陷阱三：过早优化

**症状**：
- 项目第一天就设计完整的 Harness
- 添加大量"可能需要"的约束
- Harness 比代码还复杂

**应对**：
- 从最小可行 Harness 开始
- 根据实际问题逐步添加约束
- 让 Harness 随项目成长

### 陷阱四：忽视人类判断

**症状**：
- 完全依赖 Agent
- 人类不做任何 Review
- 出现问题才发现

**应对**：
- 人类负责高层决策
- 人类做最终的批准/拒绝
- 定期 Review Agent 的工作质量

### 陷阱五：Harness 本身成为技术债务

**症状**：
- Harness 长期不更新
- 约束脚本过时
- 文档与代码不同步

**应对**：
- 定期 Review Harness
- 持续更新约束规则
- 将 Harness 维护纳入日常工作
