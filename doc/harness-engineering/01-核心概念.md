# 01 — 核心概念

> **定位**: 理解 Harness Engineering 的本质和起源  
> **来源**: OpenAI 官方博客（Ryan Lopopolo, 2026-02-11）+ 社区解读

---

## 什么是 Harness Engineering

**Harness Engineering**（驾驭工程）是一种 AI 时代的软件工程范式，核心是：

> **工程师的主要工作不再是编写代码，而是设计环境（Environment）、指定意图（Intent）、构建反馈循环（Feedback Loops）——让 AI Agent 可靠地完成编码工作。**

### 术语解析

**Harness**（驾具）这个词来自马具。在骑马语境中：

- **马** = 强大但需要引导的 AI 模型
- **缰绳/马鞍/护具** = 约束、反馈回路、文档、Linter、生命周期管理
- **骑手** = 工程师，提供方向、意图、判断力

没有 Harness，一匹强壮的马可能会把你摔下去。  
没有 Harness，一个强大的 AI Agent 可能会产出符合错误架构的代码、违反约束的实现、最终无法维护的系统。

## 为什么需要 Harness Engineering

### 传统 AI 编程的局限（"Copilot 模式"）

当前 99% 的开发者使用 AI 的方式是这样的：

```
你拆需求 → 你定方案 → AI 补代码 → 你改 Bug → 你做 CR → 你兜底
```

**AI 在这里只是"省键盘敲击次数的助理"**，核心工作仍然是人在做。

这种模式的根本问题：

1. **AI 看不见隐式规则** — 架构约束、命名规范、依赖方向这些东西从未被传达给它
2. **AI 会"忘记"** — 新对话开始时，之前的所有约定都不复存在
3. **AI 会复制坏模式** — 如果代码库有架构漂移问题，AI 会忠实地复制这个问题
4. **规则无法扩展** — 当 AI 一天产出几十个 PR 时，靠 Code Review 守住架构是不可能的

### OpenAI 的关键洞察

> "早期进展比我们预期的要慢，不是因为 Codex 能力不行，而是因为**环境规范不够明确**。Agent 缺乏完成高层目标所需的工具、抽象和内部结构。我们工程团队的首要任务变成了让 Agent 能够做有用的工作。"

**失败不是 AI 不够聪明，而是环境不够清晰。**

## Harness Engineering 的起源

### 时间线

| 时间 | 事件 |
|------|------|
| **2025 年 8 月** | OpenAI 内部团队（3人）开始实验：从空仓库开始，**所有代码由 Codex 生成** |
| **2025 年 8-12 月** | 团队逐步发现并解决 Harness 设计的核心问题 |
| **2026 年 1 月** | 产品内部上线，数百名日活用户，仓库约 100 万行代码，0 行人工编写 |
| **2026 年 2 月 11 日** | Ryan Lopopolo 发表 [《Harness Engineering》](https://openai.com/index/harness-engineering/) |
| **2026 年 2-4 月** | 技术社区广泛讨论，多个团队开始实践，社区开始形成方法论 |

### 术语在业界的扩展

值得注意的是，"Harness Engineering" 这个术语迅速从 OpenAI 的内部实践扩展为**行业通用概念**：

- `harness-engineering.ai` 网站成立，将其定义为独立的工程学科

- InfoQ、Aliyun、Juejin 等广泛报道
- 国内社区（积墨AI、Higress、掘金）大量中文解读

## 三大关键概念

### 概念一：上下文工程（Context Engineering）

> **核心问题：Agent 能"看见"什么？**

**从 Agent 的视角来看，上下文就是现实。** 任何在运行时无法获取到上下文的知识，对 Agent 来说就等于不存在。

这意味着：
- 发布在 Slack 的架构讨论 → Agent 看不见
- 写在 Confluence Wiki 的规范 → Agent 看不见
- 存在于架构师脑中的决策 → Agent 看不见
- **写进 Git 仓库的版本化 Markdown** → ✅ Agent 可以看见

**上下文工程的核心不是"写更长的 Prompt"，而是把正确的信息，以正确的方式，组织进仓库。**

### 概念二：渐进式披露（Progressive Disclosure）

不要一次性把所有信息塞给 Agent。好的 Harness 设计应该像一本好书的目录：

```
AGENTS.md (~100 行)              ← Agent 每次都会读
    ├── docs/architecture.md      ← 需要时读
    ├── docs/conventions.md       ← 需要时读
    ├── docs/design/feature-X.md  ← 处理 feature-X 时读
    └── scripts/validate.sh       ← 需要验证时执行
```

**Agent 从稳定的小入口开始，按需深入，而不是被一次性淹没。**

### 概念三：Agent 可读性（Agent Legibility）

代码库的**第一读者不再是人类，而是 Agent**。

这是一个根本性的认知转变：
- 架构设计要优先考虑 "Agent 能否理解和修改"
- 抽象层级要选 Agent 能够在上下文中完整推理的
- 依赖选择要倾向于 "无聊的技术"（boring tech）——训练集中有大量表示，行为可预测

**OpenAI 甚至宁愿自己实现某些功能的子集，而不是依赖行为不透明的第三方库。**

## Harness Engineering vs 相关概念的定位

### 三代 AI 工程范式

| 范式 | 核心问题 | 人类角色 | 局限性 |
|------|---------|---------|--------|
| **Prompt Engineering** | 如何跟模型说话？ | 精心雕琢每句指令 | 对话粒度，无法跨会话持久化 |
| **Context Engineering** | Agent 应该知道什么？ | 组装任务所需上下文 | 关注单次运行，不关注系统整体 |
| **Harness Engineering** | 整个系统如何运作？ | 设计完整运行环境 | 需要更多前期投入，学习曲线陡 |

这三者不是替代关系，而是**层层递进**的关系：

```
Prompt Engineering  ⊂  Context Engineering  ⊂  Harness Engineering
```

---

## 核心价值主张

1. **可重复性** — 不依赖某个工程师的个人技能，通过系统化方法可重复
2. **可扩展性** — 人类注意力恒定时，通过增加 Agent 并行处理来扩展
3. **自强化** — 好的 Harness 会越来越好，因为规则被机械化地执行
4. **持久性** — 知识积累在仓库里，不会因人员流动而流失

## 经典误解澄清

### 误解一："这只是把软件工程最佳实践重新包装"

**部分正确，但本质不同。**

写好文档、做好 CI、跑好测试——这些确实是传统工程实践。

关键区别在于**优化目标**：
- 传统工程：让人类读者能理解代码
- Harness Engineering：让 AI Agent 能理解、推理、修改代码

这个目标差异导致了具体实践的显著不同。例如 OpenAI 宁愿自己重新实现 `p-limit` 这样的功能，不是因为没有好用的库，而是为了**让 Agent 能完整推理这段代码的行为**。

### 误解二："这只适合 OpenAI 这种有 GPT-5 的公司"

**不正确。** Harness Engineering 的核心是**方法**，不是模型。

实验数据表明：同一模型加上精心设计的 Harness，性能提升可以相当于跨模型代际。这意味着一个普通团队用 GPT-4o + 好的 Harness，可能强过另一个团队用 GPT-5 + 糟糕的 Harness。

### 误解三："Harness Engineering 让程序员失业"

**不正确。** 它改变的是程序员做什么，而不是是否需要程序员。

工程师从"写代码的人"变成"设计让 AI 能可靠产出代码的环境的人"——这反而需要更深的系统思维、架构能力和工程判断力。
