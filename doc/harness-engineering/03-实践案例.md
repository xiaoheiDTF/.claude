## OpenAI 内部实验：从零到 100 万行

### 背景

2025 年 8 月，OpenAI 内部一个 3 人工程团队开始了一个激进的实验：

**从空仓库开始，所有代码由 Codex 生成，人类工程师不直接写任何代码。**

这不是一个玩具项目。目标是构建一个真实的内部工具，服务数百名日活用户。

### 时间线

| 时间 | 里程碑 |
|------|--------|
| **2025 年 8 月** | 3 人团队，空仓库，第一次提交由 Codex 生成 |
| **2025 年 8-10 月** | 早期进展缓慢，发现 Harness 设计的核心问题 |
| **2025 年 10-12 月** | 团队扩展到 7 人，建立完整 Harness，速度加快 |
| **2026 年 1 月** | 产品内部上线，数百名日活用户 |
| **2026 年 2 月** | Ryan Lopopolo 发表博客，分享方法论 |

### 关键数据

| 指标 | 数据 |
|------|------|
| **团队规模** | 3 人（初期）→ 7 人（后期） |
| **开发周期** | 5 个月 |
| **代码量** | ~100 万行 |
| **PR 数量** | ~1500 个 |
| **人工代码** | **0 行** |
| **日均 PR** | 3.5 个/工程师/天 |
| **开发速度** | 估计为传统手写代码的 10 倍 |

### 早期的挫折

**早期进展比预期慢。**

不是因为 Codex 能力不行，而是因为环境规范不够明确。

> "Agent 缺乏完成高层目标所需的工具、抽象和内部结构。我们工程团队的首要任务变成了让 Agent 能够做有用的工作。"  
> — Ryan Lopopolo

**具体问题**：
- Agent 不知道代码应该放在哪个目录
- Agent 不知道依赖方向是什么
- Agent 不知道如何验证自己的工作
- Agent 无法运行 UI 来验证视觉效果

### 关键突破

#### 突破一：让 Agent 能"看见"运行结果

**问题**：Agent 写了 UI 代码，但不知道渲染出来是什么样子。

**解决方案**：让 Codex 能启动 UI、读取 DOM、查看日志。

```bash
# Agent 可以执行的验证命令
npm run dev &
sleep 5
# 读取 DOM 结构
curl http://localhost:3000 | grep "data-testid"
# 查看日志
cat logs/app.log | tail -100
```

**效果**：Agent 能自己发现 UI 问题，不需要人类截图反馈。

#### 突破二：Agent 互审（Agent-to-Agent Review）

**问题**：1500 个 PR，人类无法逐一 Review。

**解决方案**：让 Agent B 自动 Review Agent A 的 PR。

```yaml
# .github/workflows/agent-review.yml
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  agent-review:
    runs-on: ubuntu-latest
    steps:
      - name: Run Agent Review
        run: |
          # 触发 Review Agent
          # 检查架构一致性
          # 检查测试覆盖率
          # 提出修改建议
```

**效果**：人类只需要做最终的"批准/拒绝"判断，大幅减少 Review 负担。

#### 突破三：6 小时长时运行任务

**问题**：复杂功能需要多步骤实现，短任务无法完成。

**解决方案**：设计支持长时运行的任务框架。

```markdown
# 长时任务示例

## 任务：实现用户认证系统

### 步骤 1：设计 API（30 分钟）
- 读取 docs/design/feature-auth.md
- 设计 REST API 接口
- 创建 API 文档

### 步骤 2：实现后端（2 小时）
- 实现 AuthService
- 实现 UserRepository
- 添加单元测试

### 步骤 3：实现前端（2 小时）
- 实现登录页面
- 实现注册页面
- 添加 E2E 测试

### 步骤 4：集成测试（1 小时）
- 运行完整测试套件
- 修复发现的问题
- 提交 PR
```

**效果**：Agent 能完成完整的功能开发，而不是只能做小修小补。

### 经验总结

> "当某件事失败时，解决方案几乎从来不是'更努力地尝试'。首要任务始终是：缺少什么能力，以及如何让它对 Agent 来说既清晰又可执行？"  
> — Ryan Lopopolo

**三个核心教训**：

1. **环境比模型更重要** — 同一个 Codex，加上好的 Harness，性能质的飞跃
2. **可见性是关键** — Agent 能"看见"的东西才能被修复
3. **机械化约束 > 文档指导** — Agent 会忘记指导，但无法绕过约束

## Stripe Minions 系统

### 背景

Stripe 是全球领先的支付公司，工程团队规模庞大，代码库复杂。他们构建了一个名为 **Minions** 的内部系统，将 Harness Engineering 应用于大规模工程实践。

### 系统架构

```
Slack 指令
    ↓
Minions 调度系统
    ↓
AI Agent 执行
    ↓
自动 Code Review
    ↓
CI/CD 验证
    ↓
自动合并 PR
```

**全流程无人值守**：从 Slack 指令到代码合并，不需要工程师手动操作。

### 规模数据

| 指标 | 数据 |
|------|------|
| **每周 PR 数量** | 数千个 |
| **人工干预** | 最小化 |
| **代码质量** | 与人工编写相当 |

### 关键设计

**Harness 设计的核心**：
- 完整的 CI/CD 管道（自动验证）
- 严格的代码规范（自动检查）
- 架构约束（自动执行）
- 测试覆盖率要求（自动验证）

**人类的角色**：
- 定义任务（Slack 指令）
- 设计 Harness（一次性投入）
- 处理异常情况（少数情况）

### 意义

Stripe 的实践证明了 Harness Engineering 的**工业化可行性**：

> 不是实验室里的玩具，而是每周处理数千个 PR 的生产系统。

## LangChain Harness 实验

### 背景

LangChain 团队做了一个对照实验：

**同一个 AI 模型，加上精心设计的 Harness，性能提升有多大？**

### 实验设计

```
对照组 A：模型 + 基本 Prompt
对照组 B：模型 + 精心设计的 Harness
```

**测试基准**：SWE-bench 2.0（软件工程基准测试）

### 实验结果

| 条件 | SWE-bench 得分 |
|------|--------------|
| **基本 Prompt** | 基准分 |
| **精心设计的 Harness** | 显著提升（相当于跨模型代际） |

**关键发现**：

> 同一模型加上精心设计的 Harness，性能提升可以相当于跨模型代际。

这意味着：
- 一个普通团队用 GPT-4o + 好的 Harness，可能强过另一个团队用 GPT-5 + 糟糕的 Harness
- **Harness 的质量比模型的能力更重要**

### LangChain Terminal Bench 2.0 实战数据

LangChain 团队进行了一个更具说服力的对照实验：

**固定模型（gpt-5.2-codex），只优化 Harness**：
- 主要改进：自动分析失败模式的工具，从 LangSmith 追踪收集失败原因并添加自验证循环
- **Terminal Bench 2.0 分数从 52.8% 跃升至 66.5%（+13.7 分）**
- **排行榜排名从约第 30 位升至约第 5 位**

**底层模型一个参数都没改。**

这一 13.7 分的提升完全来自系统提示词、工具和中间件的调优。社区因此形成了一条重要共识：

> **"在更换模型前，先检查 Harness，它通常提供最高投资回报率。"**

### 对行业的影响

这个实验改变了很多团队的思路：

- 从"等更好的模型"转向"设计更好的 Harness"
- 从"模型选型"转向"环境设计"
- 从"Prompt 优化"转向"系统架构"

## book2skills 项目

### 背景

`book2skills` 是一个开源项目，将 Harness Engineering 应用于**知识自动化传承**。

### 核心理念

**知识工厂模式**：

```
选书（人类审美）
    ↓
读书（AI 解析 PDF）
    ↓
提炼 Skill（标准模板）
    ↓
存储到 Git 仓库
    ↓
供 Agent 按需加载
```

### 工作流程

#### 步骤一：选书（人类决策）

人类工程师选择有价值的技术书籍：
- 《Clean Architecture》
- 《Designing Data-Intensive Applications》
- 《The Pragmatic Programmer》

**为什么需要人类？** 因为"哪本书有价值"需要人类的审美判断。

#### 步骤二：读书（AI 执行）

AI Agent 解析 PDF，提取关键知识：

```python
# AI 解析书籍
def extract_knowledge(pdf_path: str) -> Knowledge:
    # 读取 PDF
    # 提取关键概念
    # 识别最佳实践
    # 生成结构化知识
    pass
```

#### 步骤三：提炼 Skill（标准模板）

将知识转化为标准格式的 Skill 文件：

```markdown
# Skill: Clean Architecture

## 核心原则
1. 依赖规则：依赖方向指向内层
2. 实体层：业务规则
3. 用例层：应用逻辑
4. 接口层：适配器
5. 框架层：外部工具

## 实践指南
- 如何设计分层架构
- 如何处理依赖注入
- 如何编写测试

## 代码示例
...
```

#### 步骤四：供 Agent 使用

Agent 在处理相关任务时，自动加载对应的 Skill：

```markdown
# AGENTS.md

## 架构设计
如果需要设计分层架构，加载 skills/clean-architecture.md
```

### 意义

book2skills 展示了 Harness Engineering 的一个重要扩展：

**知识本身也可以被 Harness 化。**

不只是代码规范和架构约束，连"如何做好软件工程"的知识，都可以被编码进仓库，供 Agent 按需加载。

## Nexad 团队：自定义 Linter 的约束实战

### 背景

Nexad 团队在高 stakes 场景中早期将规则写入 `CLAUDE.md` 文档，但发现 Agent **"在压力下经常遗忘"**——尤其在 Claude 升级至 1M 上下文后，文档的约束力进一步衰减。团队 **30-40% 的时间消耗于人工质量保障**。

### 解决方案：8 个自定义 Linter 脚本

借鉴 OpenAI 的 Harness 工程实践后，团队构建了 **8 个自定义 Linter 脚本**，将业务规则编码为自动检查：

| Linter 脚本 | 检查内容 |
|------------|---------|
| `check_structured_logging.py` | 捕获 f-string 日志，强制使用结构化日志 |
| `check_soft_delete.py` | 拦截直接 `session.delete()` 调用，强制软删除 |
| `check_api_doc_sync.py` | 检测 API 变更未同步文档 |
| `check_dependency_direction.py` | 验证分层架构依赖方向 |
| `check_test_coverage.py` | 检查新代码测试覆盖 |
| `check_naming_convention.py` | 验证命名规范一致性 |
| `check_magic_strings.py` | 检测魔法字符串/数字 |
| `check_import_cycles.py` | 检测循环依赖 |

### 约束层级结构

```
约定（Convention）
    ↓
文档规则（Documented Rule）
    ↓
钩子 + 退出码（Hook + Exit Code）
    ↓
Linter / CI 检查（Mechanical Enforcement）  ← 执行效力最强
```

**核心原则**：每当新增规则，首要问题是"**能否实现为 Linter**"，若可以则优先编写 Linter 再补充文档。

这一实践深刻揭示了约束机制的核心悖论：

> **缩小解空间反而提升生产力与可靠性**——约束消除了低效的探索路径，使计算资源集中于可行方案。

## HumanLayer："成功应静默"的反馈设计

### 背景

HumanLayer 团队为 Claude Code 编写了一个 Hook 脚本，践行"成功应静默，失败才发声"原则。

### 实现细节

```bash
# 当 Agent 停止工作时自动运行
npm run format
npm run type-check

# 通过时：完全静默，零上下文污染
# 失败时：仅输出错误，触发 Harness 重新激活 Agent
```

### 效果

- 解决了 **4000 行测试输出淹没上下文**的问题
- Agent 不再对刚读到的测试文件产生幻觉
- 任务追踪准确率显著提升

## Stripe Minions 系统（补充数据）

### PR 接受率的 10 倍提升

Stripe 的预提交钩子（pre-push hooks）基于启发式运行相关 Linter，将反馈"左移"至开发阶段而非合并后。更激进的实践成果：

> **PR 接受率从 6.7% 跃升至近 70%** —— 10 倍提升源于 Harness 对工作流的系统性优化。

代理处理初始审查，仅在需要人类判断时（如新颖架构设计的批准）才升级至人工审查。

## 积墨AI 的双引擎实践

### 背景

积墨AI 是国内较早实践 Harness Engineering 的团队，他们提出了**双引擎架构**：

```
harness-creator（创建引擎）
    ↓
harness-executor（执行引擎）
```

### 双引擎设计

#### harness-creator（创建引擎）

**职责**：设计和维护 Harness 本身。

```
输入：项目需求、架构决策
输出：AGENTS.md、docs/、scripts/
```

**工作内容**：
- 分析项目特点
- 设计 Harness 结构
- 创建约束脚本
- 维护文档体系

#### harness-executor（执行引擎）

**职责**：在 Harness 环境中执行开发任务。

```
输入：功能需求、Bug 报告
输出：代码、测试、PR
```

**工作内容**：
- 读取 AGENTS.md
- 按需加载相关文档
- 实现功能
- 运行验证
- 提交 PR

### 关键洞察

> "Harness 本身也需要被维护。随着项目演进，Harness 需要更新、优化、重构。这是一个持续的工程工作，不是一次性的设置。"  
> — 积墨AI 技术博客

**双引擎的价值**：
- 分离关注点：创建 Harness vs 使用 Harness
- 专业化：不同的 Agent 专注不同的工作
- 可维护性：Harness 本身有专门的维护机制

### 实践数据

积墨AI 团队的实践结果：
- 开发速度提升 5-8 倍
- 代码质量稳定（架构约束机械化执行）
- 团队规模保持小（3-5 人）
