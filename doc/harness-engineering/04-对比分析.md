# 04 — 对比分析

> **定位**: Harness Engineering 与其他工程范式的系统性对比  
> **来源**: 社区讨论 + 理论分析

---

## vs 传统软件工程

### 核心差异

| 维度 | 传统软件工程 | Harness Engineering |
|------|------------|-------------------|
| **人类的工作** | 写代码 | 设计环境、指定意图、构建反馈循环 |
| **AI 的角色** | 不存在/辅助工具 | 执行者（Agent） |
| **质量保证** | Code Review + 测试 | 机械化约束 + 自动验证 + Agent 互审 |
| **知识载体** | Wiki、Slack、人脑 | Git 仓库（版本化、可执行） |
| **架构演进** | 重构（人工） | 持续垃圾回收（自动化） |
| **瓶颈** | 人类编码速度 | 人类注意力和判断力 |
| **文档目的** | 给人类读 | 给 Agent 读（人类可读是副产品） |
| **约束方式** | 指导（"应该怎么做"） | 机械化验证（"做得对不对"） |
| **开发速度** | 基准 | 5-10 倍 |

### 工作流对比

#### 传统软件工程

```
需求分析（人类）
    ↓
架构设计（人类）
    ↓
编码实现（人类）
    ↓
单元测试（人类）
    ↓
Code Review（人类）
    ↓
集成测试（人类）
    ↓
部署（半自动）
```

**瓶颈**：每个环节都需要人类，速度受限于人类编码速度。

#### Harness Engineering

```
需求分析（人类）
    ↓
架构设计（人类）
    ↓
环境设计（人类，一次性投入）
    ↓
编码实现（Agent）
    ↓
单元测试（Agent）
    ↓
Code Review（Agent 互审）
    ↓
集成测试（自动）
    ↓
部署（自动）
```

**瓶颈**：人类的注意力和判断力（但不是编码速度）。

### 技能要求对比

| 技能 | 传统工程师 | Harness 工程师 |
|------|----------|--------------|
| **编码能力** | 核心技能 | 仍需要，但不是主要工作 |
| **系统设计** | 重要 | 更重要 |
| **架构能力** | 重要 | 更重要 |
| **约束建模** | 较少关注 | 核心技能 |
| **反馈循环设计** | 较少关注 | 核心技能 |
| **Agent 可读性思维** | 不存在 | 核心技能 |
| **调试能力** | 核心技能 | 仍需要，但更多是调试 Harness |

### 适用场景对比

| 场景 | 传统工程 | Harness Engineering |
|------|---------|-------------------|
| **新项目** | ✅ 适用 | ✅✅ 更适用 |
| **遗留系统** | ✅ 适用 | ⚠️ 需要先梳理架构 |
| **探索期项目** | ✅ 适用 | ❌ 架构不稳定，不适合 |
| **大量重复代码** | ⚠️ 枯燥 | ✅✅ Agent 擅长 |
| **性能优化** | ✅ 适用 | ⚠️ 需要人类深度介入 |
| **创新算法** | ✅ 适用 | ⚠️ 需要人类深度介入 |

## vs Prompt Engineering

### 核心差异

| 维度 | Prompt Engineering | Harness Engineering |
|------|-------------------|-------------------|
| **优化对象** | 单次对话 | 整个系统 |
| **时间尺度** | 秒/分钟 | 月/年 |
| **持久化** | 不持久（每次重新输入） | 持久化（编码进仓库） |
| **复杂度** | 简单任务 | 复杂系统 |
| **知识管理** | 人类记忆 | Git 仓库 |
| **质量保证** | 人类检查 | 机械化约束 |
| **适用场景** | 一次性任务 | 持续开发 |

### 关系

**Prompt Engineering 是 Harness Engineering 的子集。**

```
Harness Engineering
    ├── 环境设计（仓库结构、文档、约束）
    ├── 反馈循环（CI/CD、Agent 互审）
    ├── 约束执行（Linter、测试）
    └── Prompt Engineering（如何与 Agent 沟通）
```

Harness Engineering 包含 Prompt Engineering，但远不止于此。

### 示例对比

#### Prompt Engineering 的做法

```
你：请实现一个用户注册功能，要求：
1. 使用三层架构
2. 单文件不超过 500 行
3. 使用结构化日志
4. 测试覆盖率 80% 以上

AI：好的，我来实现...

（下次对话）
你：请实现一个用户登录功能，要求：
1. 使用三层架构（又要重复一遍）
2. 单文件不超过 500 行（又要重复一遍）
...
```

**问题**：每次都要重复约束，AI 可能忘记，无法扩展。

#### Harness Engineering 的做法

```
# AGENTS.md（一次性设置）
## 架构
详见 docs/architecture/layers.md

## 代码规范
详见 docs/conventions/README.md

# scripts/validate.sh（机械化约束）
./scripts/lint-deps.sh
./scripts/lint-quality.sh
npm test
```

```
你：请实现一个用户注册功能

AI：（自动读取 AGENTS.md）
    （自动遵循架构约束）
    （自动运行 validate.sh）
    （自动提交 PR）

你：请实现一个用户登录功能

AI：（同样的流程，不需要重复约束）
```

**优势**：约束持久化，自动执行，可扩展。

## vs AI-Assisted Development（Copilot 模式）

### 核心差异

| 维度 | Copilot 模式 | Harness Engineering |
|------|-------------|-------------------|
| **AI 角色** | 辅助工具 | 执行主体 |
| **人类角色** | 写代码（AI 补全） | 设计环境（AI 写代码） |
| **工作流** | 人类主导，AI 辅助 | 人类指导，AI 执行 |
| **代码作者** | 人类（AI 帮忙） | AI（人类审核） |
| **速度提升** | 1.5-2 倍 | 5-10 倍 |
| **适用任务** | 所有任务 | 架构清晰的任务 |

### 类比

| 模式 | 类比 |
|------|------|
| **Copilot 模式** | 你开车，AI 是导航 |
| **Harness Engineering** | AI 开车，你是交通规则制定者 |

### 示例对比

#### Copilot 模式

```typescript
// 你写：
function getUserById

// Copilot 补全：
function getUserById(id: string): Promise<User> {
    return userRepository.findById(id);
}

// 你继续写：
function createUser

// Copilot 补全：
function createUser(data: CreateUserDto): Promise<User> {
    return userRepository.create(data);
}
```

**特点**：
- 你仍然在写代码
- AI 只是加速了键盘敲击
- 你仍然需要思考每一行代码

#### Harness Engineering 模式

```
你：实现用户管理功能（CRUD）

AI：（读取 AGENTS.md）
    （读取 docs/architecture/layers.md）
    （读取 docs/design/feature-user.md）
    （生成 UserService.ts）
    （生成 UserRepository.ts）
    （生成 UserController.ts）
    （生成测试文件）
    （运行 validate.sh）
    （提交 PR）

你：（Review PR，批准或拒绝）
```

**特点**：
- 你不写代码
- AI 完成整个功能
- 你只需要提供意图和判断

## vs Low-Code/No-Code

### 核心差异

| 维度 | Low-Code/No-Code | Harness Engineering |
|------|-----------------|-------------------|
| **表达能力** | 受限（预定义组件） | 完整（任意代码） |
| **灵活性** | 低 | 高 |
| **学习曲线** | 平缓 | 陡峭 |
| **适用场景** | 简单应用 | 复杂系统 |
| **技术债务** | 容易积累 | 可控（持续垃圾回收） |
| **可维护性** | 依赖平台 | 标准代码 |

### 哲学差异

| 维度 | Low-Code/No-Code | Harness Engineering |
|------|-----------------|-------------------|
| **核心思路** | 限制表达能力换取易用性 | 保持完整编程能力，提升生产力 |
| **目标用户** | 非程序员 | 程序员 |
| **输出** | 平台特定格式 | 标准代码 |

### 适用场景

```
简单 CRUD 应用
    ↓
Low-Code/No-Code ✅
Harness Engineering ⚠️（杀鸡用牛刀）

复杂业务系统
    ↓
Low-Code/No-Code ❌（表达能力不足）
Harness Engineering ✅

创新算法
    ↓
Low-Code/No-Code ❌（无法表达）
Harness Engineering ⚠️（需要人类深度介入）
```

## 三代 AI 工程范式对比

### 演进路径

```
第一代：Prompt Engineering
    ↓
第二代：Context Engineering
    ↓
第三代：Harness Engineering
```

### 系统性对比

| 维度 | Prompt Engineering | Context Engineering | Harness Engineering |
|------|-------------------|-------------------|-------------------|
| **核心问题** | 如何跟模型说话？ | Agent 应该知道什么？ | 整个系统如何运作？ |
| **优化对象** | 单次对话 | 单次任务的上下文 | 整个开发环境 |
| **时间尺度** | 秒/分钟 | 分钟/小时 | 月/年 |
| **持久化** | 不持久 | 部分持久 | 完全持久 |
| **人类角色** | 精心雕琢每句指令 | 组装任务所需上下文 | 设计完整运行环境 |
| **知识管理** | 人类记忆 | 临时组装 | Git 仓库 |
| **质量保证** | 人类检查 | 人类检查 | 机械化约束 |
| **适用场景** | 一次性任务 | 单次复杂任务 | 持续开发 |
| **局限性** | 无法跨会话 | 不关注系统整体 | 需要前期投入 |

### 包含关系

```
Harness Engineering（最大）
    ├── Context Engineering（中）
    │   └── Prompt Engineering（最小）
    ├── 环境设计
    ├── 反馈循环
    └── 约束执行
```

Harness Engineering 包含并超越了前两代范式。

## 关键洞察

### 洞察一：不是替代，是演进

Harness Engineering 不是要替代传统软件工程，而是在 AI 时代的演进。

```
传统软件工程的核心
    ↓
架构设计、质量保证、测试、CI/CD
    ↓
这些仍然重要
    ↓
但实现方式变了
```

### 洞察二：工具与围栏的平衡

| 范式 | 工具（能力） | 围栏（约束） |
|------|-----------|-----------|
| **传统工程** | 编程语言、框架 | Code Review、测试 |
| **Copilot 模式** | AI 补全 | 人类判断 |
| **Harness Engineering** | AI Agent | 机械化约束 |

**关键**：工具越强大，围栏越重要。

### 洞察三：从"有人值守"到"自动驾驶"

| 阶段 | 类比 | 特点 |
|------|------|------|
| **传统工程** | 手动驾驶 | 人类控制一切 |
| **Copilot 模式** | 辅助驾驶 | 人类主导，AI 辅助 |
| **Harness Engineering** | 自动驾驶 | AI 执行，人类监督 |

但注意：自动驾驶仍然需要道路规则、交通信号、安全系统。这就是 Harness。

## 选择指南

### 什么时候用传统软件工程？

- 团队不认可 AI 写代码
- 项目需要极致性能优化
- 创新算法研究
- 遗留系统维护（架构不清晰）

### 什么时候用 Copilot 模式？

- 所有场景都可以用（通用辅助工具）
- 不需要额外投入
- 适合个人开发者

### 什么时候用 Harness Engineering？

- 新项目从零开始
- 团队规模小但需要快速交付
- 需要生成大量重复性代码
- 架构清晰、边界明确
- 愿意投入时间设计环境

### 什么时候用 Low-Code/No-Code？

- 简单 CRUD 应用
- 非程序员用户
- 快速原型验证
