## 一句话总结

**Harness Engineering 是一种新的软件工程范式：工程师不再直接写代码，而是设计让 AI Agent 能够可靠产出高质量代码的"环境"（Harness）。**

---

## 核心类比

### 工业革命的类比

| 时代 | 力量来源 | 驾驭系统（Harness） | 人类角色 |
|------|---------|-------------------|---------|
| **工业革命** | 蒸汽机（物理力量） | 飞轮调速器、安全阀、传动系统 | 机械工程师 |
| **信息革命** | 计算机（计算力量） | 编程语言、操作系统、编译器 | 软件工程师 |
| **AI 革命** | 大语言模型（认知力量） | 架构约束、验证系统、反馈循环 | **Harness 工程师** |

蒸汽机本身不知道该驱动什么、转多快、何时停。同样，LLM 本身不知道代码应该放在哪个目录、依赖方向是什么、架构边界在哪里。

**Harness Engineering 就是为 AI 设计"缰绳、马鞍、护具"的工程学。**

## 核心转变

### 传统软件工程 vs Harness Engineering

| 维度 | 传统软件工程 | Harness Engineering |
|------|------------|-------------------|
| **人类的工作** | 写代码 | 设计环境、指定意图、构建反馈循环 |
| **AI 的角色** | 辅助（Copilot） | 执行者（Agent） |
| **质量保证** | Code Review + 测试 | 机械化约束 + 自动验证 + Agent 互审 |
| **知识载体** | Wiki、Slack、人脑 | 代码仓库（版本化、可执行） |
| **架构演进** | 重构 | 持续垃圾回收（自动化） |
| **瓶颈** | 人类编码速度 | 人类注意力和判断力 |

## 三大核心原则

### 1. 仓库即操作系统（Repository as OS）

所有架构决策、约束规则、设计文档都必须**编码到 Git 仓库**中，成为版本化、可执行的文件。

- ❌ **错误做法**: 架构约束写在 Wiki，命名规范发在 Slack
- ✅ **正确做法**: `docs/architecture.md` + `scripts/lint-deps.sh` + `AGENTS.md`

**为什么？** 因为 AI 只能"看见"仓库里的东西。不在仓库里的知识，对 AI 来说就是不存在的。

---

### 2. 地图 > 手册（Map > Manual）

`AGENTS.md` 应该是**导航地图**（~100 行），不是 1000 页的操作手册。

```markdown
# AGENTS.md（地图）

## 架构
详见 docs/architecture/README.md

## 依赖规则
详见 docs/architecture/layers.md

## 代码规范
详见 docs/conventions/README.md
```

**为什么？** 因为：
- **上下文是稀缺资源** — 巨大的指令文件会挤占任务、代码、文档的空间
- **太多指导 = 没有指导** — 当所有东西都"重要"时，就没有东西是重要的

---

### 3. 约束 > 指导（Constraints > Guidance）

不要告诉 AI "应该怎么做"，而是**机械化地验证"做得对不对"**。

| 传统做法（指导） | Harness 做法（约束） |
|---------------|------------------|
| "请遵循 Clean Architecture" | `scripts/lint-deps.sh` 检查依赖方向 |
| "单文件不要超过 500 行" | `scripts/lint-quality.sh` 自动拒绝 |
| "使用结构化日志" | `grep -r "console.log" src/` 返回非零则 CI 失败 |

**为什么？** 因为 AI 会"忘记"指导，但无法绕过机械化的约束。

## 工程师角色的转变

### 从"码农"到"环境设计师"

| 旧角色 | 新角色 |
|--------|--------|
| 写出正确的代码 | 设计让 AI 能可靠产出正确代码的环境 |
| 修 Bug | 设计让 AI 能自己发现和修复 Bug 的反馈循环 |
| Code Review | 设计让 AI 能互相 Review 的验证系统 |
| 重构 | 设计持续的"垃圾回收"机制 |

**核心技能转变**：
- ❌ 减少：手写代码、调试细节、重复劳动
- ✅ 增加：系统设计、约束建模、反馈循环设计

## 适用场景

### ✅ 适合使用 Harness Engineering 的场景

- 新项目从零开始（没有历史包袱）
- 团队规模小但需要快速交付
- 需要生成大量重复性代码（CRUD、配置、测试）
- 架构清晰、边界明确的项目
- 愿意投入时间设计"环境"而非直接写代码

### ❌ 不适合的场景

- 遗留系统改造（隐式规则太多）
- 架构尚未稳定的探索期项目
- 需要极致性能优化的底层系统
- 团队不认可"AI 写代码"的理念
- 没有时间/资源设计 Harness 基础设施

## 未来展望

Harness Engineering 仍在快速演进中。OpenAI 团队明确表示：

> "我们还不知道在完全由 Agent 生成的系统中，架构一致性如何在多年尺度上演进。我们仍在学习人类判断在哪里最有杠杆作用，以及如何编码这些判断使其能够复利增长。"

但有一点已经清晰：

**构建软件仍然需要纪律，但这种纪律更多体现在脚手架（scaffolding）而非代码本身。**
