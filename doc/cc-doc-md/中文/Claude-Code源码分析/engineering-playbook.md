# Source-Analysis / Engineering-Playbook

> 来源: claudecn.com

# 工程落地清单

## 为什么需要这页

看懂源码结构，不等于能把这些能力装进自己的团队流程。

真正落地时，团队通常会卡在这些问题上：

- 先做命令、skills、hooks、插件还是 MCP？
- 哪些能力应该先开放，哪些必须先收紧？
- 怎么把“会用一个功能”变成“能稳定跑一条工作流”？
- 怎么避免目录越来越多、规则越来越散、最后没人维护？
这页的目标是把源码分析里的结论，压缩成一条更接近工程执行的路线。

## 阶段一：先做能力目录，不急着做平台

第一步不是“搭一个完整框架”，而是先确定你到底需要哪一层能力。

建议按这五层做最小盘点：

| 层次 | 适合解决的问题 | 先看哪里 |
| --- | --- | --- |
| **命令入口** | 让用户以固定动作触发能力 | [命令表面](https://claudecn.com/docs/source-analysis/commands/) |
| **工具能力** | 让模型能可靠操作文件、终端、外部系统 | [工具平面](https://claudecn.com/docs/source-analysis/tools/) |
| **执行边界** | 控制什么时候能写、什么时候必须问、什么时候必须停 | [权限治理](https://claudecn.com/docs/source-analysis/governance/) |
| **扩展容器** | 把 skills、命令、hooks、MCP 打包成一组能力 | [插件系统](https://claudecn.com/docs/source-analysis/plugins/) |
| **长期连续性** | 让上下文、记忆、恢复和回滚形成闭环 | [记忆系统](https://claudecn.com/docs/source-analysis/memory/) |

### 本阶段产物

- 一张你自己的“能力目录表”
- 明确哪些能力是项目级，哪些是用户级，哪些是运行时临时能力
- 只选一到两条最短路径先做，不要一开始把所有层都铺开
## 阶段二：先收紧执行边界，再谈自动化

很多团队一开始最容易做反的一点，是还没建立权限边界，就急着做自动化。

更稳的顺序是：

- 先明确哪些动作只读
- 再明确哪些动作需要审批
- 再决定哪些动作可以自动执行
- 最后才把 Hook、MCP、插件和多代理串起来
这里最值得借鉴的，不是某个具体配置，而是 Claude Code 的边界观：

- plan 不是提示词，而是权限收紧
- sandbox 不是可选项，而是最后一道边界
- hook 不是“顺手做点事”，而是执行链上的判定节点
- 自动化只有在可回退时才值得开启
### 本阶段检查项

- 是否先有 研究方法 中说的事实边界？
- 是否能明确区分只读、写入、外部访问三类动作？
- 是否存在“失败后退回安全状态”的路径？
- 是否能回答“谁批准、在哪批准、失败后怎么恢复”？
## 阶段三：把能力组合成最小工作流

功能不会自己变成工作流。真正进入团队使用前，至少要先做出一条最小闭环。

一条可用的最小闭环通常长这样：

- 用户从固定入口发起任务
- 系统进入分析或规划阶段
- 工具按边界执行动作
- 独立验证结果
- 保留回滚或恢复路径
这时要重点看三类组合：

| 组合 | 典型用途 | 对应主题 |
| --- | --- | --- |
| **命令 + 工具 + 权限** | 日常固定动作、低风险自动化 | [命令表面](https://claudecn.com/docs/source-analysis/commands/) + [工具平面](https://claudecn.com/docs/source-analysis/tools/) + [权限治理](https://claudecn.com/docs/source-analysis/governance/) |
| **记忆 + 压缩 + 会话恢复** | 长任务、多轮任务、上下文续跑 | [记忆系统](https://claudecn.com/docs/source-analysis/memory/) + [运行时流程](https://claudecn.com/docs/source-analysis/runtime/) |
| **多代理 + 独立验证** | 复杂任务拆解、并行调研与验证 | [Coordinator 编排](https://claudecn.com/docs/source-analysis/coordinator/) |

### 本阶段产物

- 一条“从输入到验证”的最小工作流
- 一个明确的验证节点，而不是让实现者自己判断“已经完成”
- 一条能演示给团队看的最小示例
## 阶段四：把工作流变成团队资产

当最小工作流跑通后，才值得进入“如何共享和分发”。

此时应该回答的是：

- 这套能力是项目资产还是个人资产？
- 是应该做成命令模板，还是 skill，还是插件？
- 配置、资源和脚本是跟仓库走，还是跟用户环境走？
Claude Code 源码里最值得借鉴的点是：**不要让用户手动拼装一大堆零件。**

如果一套能力必须同时依赖命令、Hook、MCP、skills 和设置项，就应该把它收敛成一个更清晰的分发单元，而不是让团队成员各自复制粘贴。

### 本阶段建议

- 单一动作优先做成命令或 skill
- 一组协作能力再考虑做成插件容器
- 团队共享内容和个人本地内容必须分层
- 所有入口都要有 README 或索引页，避免“装了但没人知道怎么用”
## 阶段五：最后补运行保障

真正让系统长期可维护的，不是“功能多”，而是运行时是否有保障。

至少要补齐这五类保障：

- 恢复：失败后能否继续、重试、回滚
- 会话连续性：长任务中断后能否续跑
- 成本观察：token、时间、模型切换是否可见
- 状态外化：关键记忆和计划是否只存在于 transcript
- 验证独立性：有没有人或机制专门负责验收，而不是实现顺手验收
对应阅读建议：

- 运行时流程
- 记忆系统
- 成本追踪
- Hooks 韧性
## 常见误区

### 误区一：先做完整框架

很多团队还没跑通一条最小路径，就先建了大量目录、模板和规范。结果是结构先复杂，能力反而没有闭环。

### 误区二：把计划模式当成“多一步确认”

计划模式真正有价值的地方，不是多问一句，而是把“探索、审批、执行”拆成可治理的状态机。

### 误区三：只有实现，没有独立验证

如果实现者和验证者共享同一上下文，系统很容易高估自己的正确性。这也是 [Coordinator 编排](https://claudecn.com/docs/source-analysis/coordinator/) 最值得迁移的地方。

### 误区四：把插件当成“大号技能”

插件解决的是分发、安装、验证、更新和卸载，不只是“把几个文件放到一起”。

## 推荐顺序

如果你准备把 Claude Code 的能力迁移进自己的团队流程，建议按这个顺序推进：

- 读 研究方法
- 做能力目录盘点
- 先建立执行边界
- 跑通一条最小工作流
- 再做团队分发和运行保障
这比直接模仿别人的目录结构，更接近可长期维护的工程路线。
