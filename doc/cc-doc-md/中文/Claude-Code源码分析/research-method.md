# 研究方法

## 为什么要有这页

如果只读官方文档，你会知道 Claude Code 有哪些能力、推荐怎么用；如果只读源码快照，你会看到大量路径、类型和运行时细节，但很容易失去阅读顺序。

这页的作用不是重复官方说明，而是回答三个更实际的问题：

- 先看什么，才能最快建立正确心智模型？
- 什么信息适合当作事实边界，什么只适合当作结构信号？
- 从别人的示例和经验里，真正值得迁移到自己团队里的是什么？
## 和官方资料的分工

官方资料更适合回答这些问题：

- 这个功能是什么
- 官方支持的边界在哪里
- 最小示例怎么跑通
- 当前推荐的接入方式是什么
源码分析更适合回答这些问题：

- 这些能力在系统里如何会合
- 为什么权限、压缩、记忆和恢复会被放在主循环里
- 哪些设计是稳定骨架，哪些只是局部信号
- 如果你要自己做 Agent Harness，哪些原则值得迁移
换句话说，**官方文档定义能力边界，源码分析解释系统结构。**

## 我们提炼出的四条方法

### 1. 先确认能力原语，再看组合方式

最新的官方 quickstarts、skills 与插件资料都有一个共同点：先给出能力原语和最小可运行入口，再逐步扩展。

因此在阅读 `source-analysis` 时，建议顺序不是“从目录开始”，而是：

- 先看 运行时流程 和 工具平面
- 再看 命令表面 与 插件系统
- 最后进入 Coordinator 编排 与 MCP 与 Bridge
这样读，你看到的是“能力如何被组织”，而不是“文件如何被堆放”。

### 2. 不按目录抄结构，要按任务重组阅读

社区资料里最有价值的部分，不是目录长什么样，而是它们普遍强调：

- 要有阅读路线
- 要按任务和阶段理解能力组合
- 要知道不同阶段该学什么、不该急着学什么
这也是我们把 `source-analysis` 重组为“总览 / 结构层 / 运行层 / 能力层 / 演化层”的原因。目录树只是证据入口，不应该直接变成内容结构。

### 3. 一边替换，一边保留验证点

很多资料都会给出最小示例，但真正决定能不能进入生产的，不是“先跑起来”，而是“每替换一步都还有验证点”。

因此阅读源码时要重点盯住这些位置：

- 权限判定链是否可回退
- compact 触发后能否恢复主循环
- 多代理编排里验证是否独立
- 插件、skills、MCP 是否按需暴露，而不是一次性敞开
这也是为什么我们在源码分析里反复强调“运行时不变量”，而不只是罗列模块。

### 4. 计划先行，比直接执行更重要

最新资料里一个非常明显的趋势是：成熟的 Agent 系统越来越重视“先探索、先规划、再执行”。

这不是单纯的交互优化，而是工程边界控制：

- 先让用户确认方向
- 再开放写权限
- 把计划作为显式工件沉淀
- 让恢复、回滚和协作都有落点
因此，读 Claude Code 源码时，不能只把 Plan Mode 看成一个功能点，它更接近一套“人机对齐 + 权限收敛 + 任务持久化”的状态机。

## 三种推荐读法

### 如果你是普通用户

先看：

- 运行时流程
- 权限治理
- 成本追踪
目标是理解系统为什么会拒绝、压缩、恢复，而不是背功能清单。

### 如果你是 Agent 构建者

先看：

- 架构地图
- 运行时流程
- 记忆系统
- Coordinator 编排
目标是提炼你自己的运行时骨架、治理边界和验证方式。

### 如果你是源码研究者

先看：

- 源码分析总览
- 工具平面
- 命令表面
- 扩展与信号
目标是区分直接证据、语义聚类和解释性判断，不把快照信号误读成产品承诺。

## 最值得迁移到自己系统里的，不是目录而是原则

我们从最新资料中真正吸收的，不是别人的写法和仓库长相，而是这些更稳定的工程原则：

- 用最小能力原语建立入口
- 用任务路径而不是文件路径组织内容
- 在替换示例时保留验证点
- 让计划、权限、恢复与回滚形成闭环
- 把“验证必须独立”当作系统约束，而不是事后补丁
如果你带着这些原则回头看 Claude Code，就更容易分清哪些是产品外观，哪些才是 Agent Harness 的骨架。
