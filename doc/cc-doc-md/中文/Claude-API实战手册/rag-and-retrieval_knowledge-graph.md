# Knowledge Graph（知识图谱构建）

当问题的重点从“找相似内容”变成“理解实体之间的关系”时，知识图谱往往比单纯的向量检索更合适。它不是为了让系统显得更复杂，而是为了让信息之间的连接关系可以被保留下来、验证和复用。

## 什么时候该考虑知识图谱

- 你关心的是实体之间的显式关系
- 你需要多跳推理，而不是只找最像的片段
- 你希望结果能被追踪、解释和复核
如果你的问题仍然主要是“把相关资料找出来”，基础 RAG 或上下文化检索通常更直接。

## 一条典型工作流

知识图谱类任务通常会经历四步：

- 识别实体
- 抽取关系
- 合并同义或重复实体
- 在图结构上做查询、汇总或解释
真正有价值的部分，不是把图做出来，而是让这些步骤能稳定复现，并且能回答你原本想解决的问题。

## 它和向量检索的差别

### 向量检索更擅长

- 找到语义相近的内容
- 在大段文本里快速召回相关片段
- 用较低门槛先做可用系统
### 知识图谱更擅长

- 保留实体与关系的结构
- 处理多跳关系和关系链追踪
- 支持更强的解释性和审计性
## 更合适的使用方式

最常见的高效路径不是“一开始就做图谱”，而是先用基础检索确认任务成立；当你发现问题已经不再是“找不到内容”，而是“内容之间关系说不清楚”，再进入图谱化。

## 常见坑

### 为了复杂而复杂

如果没有明确的关系型问题，知识图谱只会增加系统负担。

### 实体边界定义不清

如果实体命名、合并规则和关系类型本身就不稳定，后面的图结构很难可靠。

### 没有质量验证

图谱系统很容易“看起来很聪明”，但真正的问题是：关系是否正确、召回是否稳定、结果是否可解释。没有评测，往往很难知道它到底有没有变好。
