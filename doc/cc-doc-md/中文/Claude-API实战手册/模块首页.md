# Cookbook

Cookbook 更适合“边看边跑、边跑边改”的学习方式。它不是按概念层层讲解的教程，而是一组围绕真实问题展开的可运行示例：你先挑一个最接近当前任务的例子，跑通它，再把数据、工具和约束换成自己的版本。

如果你还不确定自己现在更该读 `Claude Code`、`Quickstarts` 还是 `Cookbook`，先看 [学习路径](https://claudecn.com/docs/learning-paths/) 会更省时间。

## 这部分最适合什么场景

- 你想快速验证一个能力模式是否可行
- 你想比较不同实现路径，而不是只看抽象原则
- 你已经有产品想法，需要一个可修改的起点
- 你想把实验结果逐步收敛成可重复的工程做法
## 推荐的使用方式

### 先从一个问题出发
不要先从“我要看完整示例库”开始，而是先从当前问题出发：检索、工具调用、Agent 编排、多模态、输出控制，还是评测与回归。

### 先跑最小示例，再做替换

第一轮目标是看懂核心链路，而不是一下子把所有参数都改成自己的。先确认示例能稳定运行，再替换数据、提示词、工具和边界条件。

### 一边改，一边留验证点

只要进入“我要把示例变成自己的功能”这一步，就应该同时准备最小评测样本、断言或回归检查。否则你很容易把示例改到能跑，却不知道质量有没有变差。

## 开始前准备

### 基础环境

- Python 3.11+
- 可用的 Anthropic API Key
- 一份安全、可重复的测试数据
### 最常见的准备命令

```bash
uv sync --all-extras
cp .env.example .env
make test-notebooks
```

如果你准备执行真实 Notebook，再补一条：

```bash
make test-notebooks-exec
```

Notebook 运行很容易把“实验成本”和“真实业务数据风险”同时带进来。先用小样本、脱敏数据和测试环境。

## 一个简单但重要的原则
示例的价值不在于“替你决定答案”，而在于“给你一个可以快速验证和不断收敛的起点”。读 Cookbook 时，最有效的问题通常不是“它还能做什么”，而是“我手头这个问题，最该先借哪条路径来验证”。
