# Quickstarts

Quickstarts 最有价值的地方，不是“把 demo 跑起来”，而是让你更快选对**产品起点**。如果你第一次把 Claude 接进真实应用，或者你已经有明确的产品目标但还没有最小骨架，Quickstarts 是最值得优先看的官方资料之一。

如果你现在还不确定自己该先从 `Quickstarts`、`Cookbook` 还是 `Claude Code` 进入，先看 [学习路径](https://claudecn.com/docs/learning-paths/)。

## 先按目标选起点

### 做对话型产品
如果你的重点是问答、客服、知识助手或内部支持，先看 `Customer Support Agent`。它最接近“对话 + 检索 + 回复组织”的真实产品骨架。

### 做分析型产品

如果你要处理表格、指标、趋势或图表，先看 `Financial Data Analyst`。它更适合验证“用户一句话提出问题，系统返回图表和解释”的闭环。

### 做网页自动化

如果你的目标是网页导航、表单填写、信息提取或浏览器里的重复操作，先看 `Browser Automation Demo`。这里最值得学的是 DOM 感知动作，而不只是“能不能点到按钮”。

### 做桌面操作自动化

如果你的任务发生在桌面应用、系统界面或跨窗口流程里，先看 `Computer Use Demo`。这类项目的关键不只是操作能力，更是隔离、安全和确认链路。

### 做能持续推进的编程智能体

如果你关心的是“连续几轮甚至多次会话都能继续推进一个开发任务”，先看 `Autonomous Coding Agent`。它更贴近任务拆解、阶段推进和结果持续迭代。

### 先理解 Agent 基础能力

如果你还在建立 Agent 心智模型，先看 `Agents Reference`。这样后面再看自动化或[ 编程](#)类项目时，就不会只停留在“跑起来”，而能看懂为什么这样设计。

## 不要直接照抄 Quickstarts

Quickstarts 更适合作为“官方产品骨架参考”，而不是最终站内内容。更合理的提取方式是：

- 先抽出每个项目要解决的真实问题
- 再抽出它的最小闭环：输入、模型、工具、输出
- 把工程细节重写成“上线前要补什么”
- 最后再保留必要的官方代码入口和阅读路径
这样做的好处是，读者读完一页就能判断“这是不是我该先看的项目”，而不是被 README 细节淹没。

## 更高效的使用方式

### 先选一个和目标最接近的起点

不要一开始横向对比所有示例。先挑一个与你的产品目标最接近的起点，把它当作最小可运行骨架。

### 第一轮只理解 4 件事

- 用户输入从哪里进入
- 模型输出在哪里生成
- 工具或数据源怎样接入
- 错误和边界情况怎样处理
### 第二轮再做替换

建议一次只改一个维度：提示词、工具、数据源、界面或权限控制。这样你更容易判断“到底是哪一层带来了效果变化”。

### 第三轮补生产必需项

当交互和主流程稳定后，再补鉴权、日志、重试、监控、人工兜底和评测。很多项目不是卡在模型能力，而是卡在这些工程细节。

## 开始前准备

### 基础环境

- 可用的 Anthropic API Key
- Node.js LTS 或 Python 3.11+（取决于起点技术栈）
- 一份可反复测试的演示数据
### 最常见的环境变量

```bash
export ANTHROPIC_API_KEY="your-api-key"
```

如果示例包含浏览器或桌面自动化，建议优先在沙箱环境、测试账号或隔离浏览器配置里验证，不要直接在真实业务环境里试错。

## 推荐阅读路径

### 初次接入 Claude API

先从 `Customer Support Agent` 或 `Financial Data Analyst` 入手，目标是理解“用户请求 -> 模型处理 -> 应用响应”的基本链路。

### 想做自动化工具

先看 `Browser Automation Demo`，再看 `Computer Use Demo`。这样更容易先掌握结构化页面操作，再进入更开放的屏幕与系统交互。

### 想做 Agent 或长期任务系统

先用 `Agents Reference` 建立工具循环和状态管理基础，再进入 `Autonomous Coding Agent` 这类更长链路的任务。

## 上线前别跳过的检查

- 是否对密钥、Cookie、用户数据做了隔离
- 是否有失败重试与超时处理
- 是否记录了关键操作日志
- 是否对高风险动作加了人工确认
- 是否为核心任务准备了最小评测样本
## 继续阅读

- Cookbook - 按问题类型拆开的可运行示例
- Claude Code - 终端里的开发与自动化工作流
- Agent Skills - 把稳定做法沉淀成可复用能力
- Claude API 文档 - 官方接口与模型能力说明
## 常见问题

### 我应该先追求“示例能跑”，还是先追求“结构最优”？

先跑通，再收敛结构。对大多数团队来说，先看到完整链路，远比一开始就设计出最完美的抽象更重要。

### 可以直接把示例拿去上线吗？

更合理的做法是把它当成产品原型或实现骨架。真正上线前，仍然需要补安全、监控、错误处理、评测和权限治理。

### 如果我还不确定做什么应用，应该从哪里开始？

优先选最贴近你业务目标的那一类：对话、分析、网页自动化、桌面自动化或长任务 Agent。先解决“场景相似”问题，再考虑“技术最全”。
