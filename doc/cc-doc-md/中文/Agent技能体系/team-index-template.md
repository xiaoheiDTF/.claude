# 团队索引页模板

很多团队的问题不是“不会写 Skill”，而是 Skill 越写越多以后，没有一页真正给人看的入口。

结果通常会变成这样：

- 新成员不知道有哪些 Skills
- 老成员只记得自己常用的几个
- 同一个任务每个人启用的组合都不一样
- Skills 明明写了，但团队 adoption 很低
解决方式并不复杂：给 `.claude/skills/README.md` 或团队 Skills 目录写一页真正面向人的索引。

这页的目标不是解释 Skill 规范，而是帮助团队快速回答三个问题：

- 我们现在有哪些 Skills？
- 各自适合什么场景？
- 一个任务通常应该先启用哪几个？
如果你还没写第一个 Skill，先看 [快速开始](quickstart/)。如果你想先把 Skill 写得更像样，再看 [最佳实践](best-practices/)。

## 一页团队索引应该至少解决什么

一个有效的团队索引页，至少要同时服务两类人：

- 第一次进项目的人：快速知道这里有哪些 Skills
- 已经在用的人：快速决定当前任务该用哪几个组合
所以这页至少要有四块内容：

- 分类列表
- 每个 Skill 的一句话用途
- 常见任务的推荐组合
- 维护规则或更新提示
## 最小模板

下面这个模板已经足够让大多数团队开始使用：

```markdown
# Team Skills Index

本目录收录本项目/团队长期维护的 Skills。

## 使用原则

- 先看任务类型，再选 Skill
- 一个任务优先启用 1 到 3 个核心 Skills，不要一次全开
- 发现过期内容时，先修索引，再修 Skill 主体

## Skills by Category

### 开发与质量

| Skill | 作用 | 何时使用 |
|---|---|---|
| `testing-patterns` | 统一测试写法与 TDD 节奏 | 新增测试、补回归、修 flaky test |
| `code-review-checklist` | 固定审查清单 | 提交前自查、PR 复审 |
| `systematic-debugging` | 结构化排障 | 构建失败、线上问题回溯 |

### 前端与交互

| Skill | 作用 | 何时使用 |
|---|---|---|
| `frontend-design` | 统一前端界面实现方向 | 新页面、新组件、视觉改版 |
| `core-components` | 复用设计系统组件 | 现有组件库内开发 |

## 推荐组合

### 新增功能
1. `backend-patterns`
2. `testing-patterns`
3. `code-review-checklist`

### 修复线上问题
1. `systematic-debugging`
2. `testing-patterns`

### 新页面开发
1. `frontend-design`
2. `core-components`
3. `testing-patterns`

## 维护规则

- 新增 Skill 后，必须同步更新本索引
- 每个 Skill 只写 1 句用途，不写成长文
- 每月抽查一次失效条目和过期组合
```

## 写团队索引时最容易忽略的 4 件事

### 1. 不要把索引写成“另一本文档”
索引页不是再写一遍所有 Skill 的完整说明。它只需要帮助用户快速做选择。

所以每个 Skill 最好只写：

- 名称
- 1 句话用途
- 1 句触发场景
如果展开成大段原理说明，这页很快就会失去入口价值。

### 2. 一定要写“推荐组合”

很多团队只列 Skill 名单，但不写组合。这会导致：

- 大家知道有哪些 Skill
- 但不知道某个任务该先用哪几个
“组合入口”通常比“完整列表”更重要，因为真实工作不是单个 Skill 的问题，而是一串任务。

### 3. 类别不要按实现方式分，尽量按任务分

比起按“文档类 / 脚本类 / 分析类”分组，更推荐按团队真实任务分组，比如：

- 开发与质量
- 前端与交互
- 数据与 API
- 文档与沟通
- 平台与运维
因为成员真正关心的是“我要做什么”，而不是“这个 Skill 在实现上属于哪一类”。

### 4. 索引必须跟变更一起维护

一个很常见的问题是：

- Skill 已经加了
- 但索引没更新
两个月后，索引就不可信了。

所以最实用的规则是：

- 新增 Skill：必须同时更新索引
- 重命名或废弃 Skill：必须同步调整索引
- 定期抽查索引里的组合是否仍然有效
## 一个更适合团队的写法

如果你的团队 Skill 已经比较多，可以用“两层入口”：

### 第一层：快速选择

- 新增功能
- 修 bug
- 代码审查
- 新页面
- 文档编写
- 数据处理
### 第二层：分类索引

- 列出全部 Skills
- 每个 Skill 只保留一句说明
这种写法的好处是：

- 新成员先按任务进入
- 老成员需要查全量时也有索引可用
## 什么时候这页应该继续升级

如果你开始出现下面这些情况，说明团队索引页值得再升级：

- Skills 超过 10 到 15 个
- 团队成员开始频繁问“这个任务该用哪个 Skill”
- 不同成员对同一类任务使用方式差异很大
- 你开始希望把某些组合入口做成命令、agent 或插件
这时你可以继续加：

- 任务到 Skill 的映射表
- 项目内强制推荐组合
- 与 /command、agents、hooks 的联动说明
## 推荐放置位置

常见放法有两种：

- .claude/skills/README.md
- 项目级文档中的 skills-index.md
如果目标是给 Claude 和团队成员共用，优先放在 `.claude/skills/README.md`。

如果目标是给更广泛的项目成员阅读，也可以在项目文档中保留一份可读性更强的版本。
