# 选择插件来源

很多人第一次接触插件时，真正困惑的不是“命令怎么写”，而是“我应该从哪里开始”。对大多数团队来说，这个问题的答案并不复杂：先看官方目录，再判断它是否已经足够；如果不够，再进入团队插件或本地插件。

## 为什么很多人先从官方目录开始

官方目录的优势不是“包办一切”，而是更容易开始：

- 安装路径清晰
- 常见能力更容易找到
- 更适合先验证工作流是否成立
如果你现在的目标只是提升编码、审查、提交、设计检查、语言服务或常见工具接入效率，先从官方目录开始通常成本最低。

如果你已经确认要从官方目录开始，但还不知道先装哪一类，建议直接看：[按用途选择插件](by-use/)。

## 你通常能在这里找到什么

最常见的能力大致分成几类：

- 开发工作流：审查、精简代码、提交辅助、特性开发
- 语言与工具接入：常见语言服务、本地工具增强
- 外部服务连接：代码托管、协作工具、平台服务
- 输出风格与辅助能力：解释风格、学习模式、特定工作方式
这意味着官方目录更适合做“第一层增强”：先让常见任务更顺，而不是一上来就处理所有组织定制需求。

## 什么时候官方目录已经够用

如果你的需求属于这些类型，通常可以先停在官方目录：

- 常见编码工作流
- 常见平台集成
- 通用的审查、提交、排障和输出增强
## 什么时候该转向团队插件

如果你的需求明显带有组织特征，例如：

- 团队自己的代码规范
- 固定的评审清单
- 内部平台入口
- 私有 MCP 服务
- 需要与公司流程强绑定的命令
这类场景更适合团队插件，而不是继续在通用目录里寻找替代品。

## 什么时候直接用本地插件更合适

当你还在验证想法、快速试错，或者只是想给当前项目加一个临时能力时，本地插件通常最快。它更适合短周期实验，不必一开始就考虑共享和分发。

## 安装前建议检查的 4 件事

- 维护者是谁
- 需要哪些权限、工具或二进制依赖
- 这套能力解决的是不是你当前最高频的问题
- 装上以后是否便于团队维护
## 一条简单决策线

- 先问自己：这是通用需求，还是团队特有需求？
- 通用需求先从 claude-plugins-official 开始
- 团队特有需求优先考虑团队插件
- 仍在实验阶段，就先用本地插件
## 最短使用路径

```bash
/plugin marketplace list
/plugin install code-simplifier@claude-plugins-official
/plugin list
```

如果你下一步要做的是安装和验证，继续看 [发现和安装插件](discover-plugins/)。如果你已经确认现成能力不够，继续看 [创建插件](create-plugins/)。
