# 上下文角色：开发 / 调研 / 审查

Claude Code 的“回答质量”很大程度取决于你给它的任务角色。社区实践里常用的做法是：在不同阶段显式切换上下文角色，把输出结构固定下来，减少跑偏。

这里给出三种最常用的角色模板（来自社区配置仓库的思路，已按中文语境整理），你可以直接复制到对话里当作“开场白”。

## 1) 开发（Dev）：先做出可跑版本

适用场景：

- 功能实现、修 Bug、写脚本、补测试
- 你希望 Claude “先落地、后解释”
建议指令（可复制）：

```text
上下文：开发模式
目标：先把功能做出来并可验证
行为：
- 先写代码，后解释
- 修改后运行最小验证（测试/构建/脚本）
- 变更保持原子、避免无关重构
```

## 2) 调研（Research）：理解优先，先给证据
适用场景：

- 读陌生代码库、排查复杂问题根因
- 需要先收集证据，再做结论
建议指令（可复制）：

```text
上下文：调研模式
目标：先理解再行动
行为：
- 先阅读/搜索再下结论
- 先提出澄清问题与假设
- 用证据验证假设
输出顺序：发现（证据）→ 推论 → 建议
```

## 3) 审查（Review）：按严重程度给可执行建议
适用场景：

- PR 审查、代码质量与安全审查
- 你希望输出“按优先级排序”的行动清单
建议指令（可复制）：

```text
上下文：审查模式
目标：质量 + 安全 + 可维护性
行为：
- 先通读再评论
- 按严重程度排序：CRITICAL > HIGH > MEDIUM > LOW
- 提供可执行修复建议（不仅指出问题）
输出格式：按文件分组 + 每条包含定位信息
```

## 什么时候切换角色
一个稳定的节奏是：

- 需求不清晰 / 代码陌生：先用“调研”
- 方案不确定 / 影响面大：切到 Plan Mode 做规划（见 计划模式）
- 开始写代码：切到“开发”
- 改动完成准备合并：切到“审查”
如果你想把这套做法工程化（写进 `CLAUDE.md`、固化成 `/command` 或 Hook），可以从 [团队 Starter Kit](https://claudecn.com/docs/claude-code/advanced/starter-kit/) 开始。
