# 高效实践

掌握高级技巧可以显著提升使用 Claude Code 的效率。本节介绍处理复杂任务的策略和常见场景的最佳实践。

## 使用检查清单

对于多步骤任务，让 Claude 创建检查清单：

```
> 将需要修复的 lint 错误写入 TODO.md 作为检查清单
```

生成的文件可能是：

```markdown
## Lint 错误修复清单

- [ ] src/auth/login.ts:42 - 未使用的变量 'temp'
- [ ] src/api/users.ts:15 - 缺少返回类型注解
- [ ] src/utils/format.ts:8 - 使用了 'any' 类型
```

然后逐个处理：

```
> 修复清单中的第一个问题
```

## 使用草稿本
对于需要迭代的任务，使用 Markdown 文件作为草稿本：

```
> 创建 SCRATCH.md，在里面规划 API 重构方案
```

```
> 更新 SCRATCH.md 中的方案，添加错误处理部分
```

草稿本的优势：

- 保留思考过程
- 方便回顾和修改
- 可以分享给团队成员
- 不会污染实际代码
## 结合 Escape 和"记忆"

中断后让 Claude 记住关键信息：

```
> [按 Escape]
> 记住：这个项目使用 Tailwind CSS，不要用行内样式
```

```
> 把这条规则添加到 CLAUDE.md
```

这样 Claude 在后续任务中会自动遵循这些规则。

## 战略性压缩（compact）

当一个会话跨越多个阶段（调研 → 设计 → 实现 → 验证）时，上下文往往会越来越“浑”。与其等自动压缩在随机时刻发生，不如把压缩当作“阶段切换”的可控动作（如果你的版本支持 `/compact` 或类似功能）。

更工程化的做法是配合 Hooks 做“会话交接”和“压缩标记”。详见：[会话连续性与战略压缩](https://claudecn.com/docs/claude-code/workflows/session-continuity/)。

## 批量处理 vs 逐个处理

**批量处理适合：**

- 相同模式的重复任务
- 简单的查找替换
- 格式化和规范化
```
> 将所有 console.log 替换为 logger.debug
```

**逐个处理适合：**

- 需要判断的复杂修改
- 不确定是否正确的更改
- 学习代码库时
```
> 逐个审查每个 TODO 注释，告诉我应该如何处理
```

## 常见场景工作流

### 修复 Bug

```
> 1. 描述问题：用户登录后被踢出
> 2. 让我先看看相关代码
> [Claude 分析代码]
> 3. 你觉得问题出在哪里？
> [Claude 给出诊断]
> 4. 修复它
> 5. 添加测试确保不会再次出现
```

**关键点：**

- 先理解再修复
- 让 Claude 解释根本原因
- 添加回归测试
### 添加新功能

```
> /plan 添加用户头像上传功能

> [审查规划后]
> 按计划执行第一步

> [确认后继续]
> 继续下一步
```

**关键点：**

- 使用规划模式
- 分步执行
- 每步确认
### 代码重构

```
> 分析 UserService 类，列出重构建议

> [审查建议]
> 执行建议 #2：提取验证逻辑到单独的类

> 确保所有测试仍然通过
```

**关键点：**

- 先分析后行动
- 一次只做一个重构
- 保持测试绿色
### 编写测试

```
> 分析 calculatePrice 函数的边界情况

> 为每个边界情况编写测试

> 运行测试确认覆盖率
```

**关键点：**

- 让 Claude 识别边界情况
- 生成全面的测试用例
- 验证覆盖率
## 效率倍增技巧

### 并行任务

在一个终端中让 Claude 执行耗时任务时，可以开另一个终端处理其他任务：

```bash
# 终端 1
claude "运行完整测试套件并分析失败"

# 终端 2
claude "同时审查 PR #123 的代码"
```

### 模板化常用指令
创建 shell 别名：

```bash
alias cr='claude "审查当前分支的所有更改并提出建议"'
alias ct='claude "运行测试并修复失败的用例"'
alias cc='claude commit'
```

### 使用上下文文件
为不同项目维护不同的 CLAUDE.md：

```
project/
├── CLAUDE.md           # 项目级别
├── frontend/
│   └── CLAUDE.md       # 前端特定规则
└── backend/
    └── CLAUDE.md       # 后端特定规则
```

## 避免常见陷阱

| 陷阱 | 解决方案 |
| --- | --- |
| 一次性给太多任务 | 分解为小步骤 |
| 不审查生成的代码 | 始终 review 后再提交 |
| 忽略测试 | 每次修改后运行测试 |
| 模糊的需求描述 | 提供具体的预期行为 |
| 不使用 /clear | 定期清理上下文 |

## 效率检查清单
高效使用 Claude Code 的日常习惯：

- 项目开始时运行 /init
- 维护最新的 CLAUDE.md
- 复杂任务先规划
- 及时使用 Escape 纠正方向
- 定期使用 /clear 清理上下文
- 代码修改后运行测试
- 使用检查清单跟踪多步骤任务
