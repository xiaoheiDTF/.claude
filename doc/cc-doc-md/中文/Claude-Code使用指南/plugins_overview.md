# 先理解插件

插件的核心价值，不是把一句提示词换个包装，而是把一套稳定能力打包成可安装扩展。它可以同时带上命令、Agents、Skills、Hooks、MCP 配置，甚至项目常用工具的接入方式。

## 先建立一个正确的心智模型

把插件理解成“能力包”会更准确。它的重点不是单个文件，而是把一类工作方式变成能复用、能共享、能更新的扩展。

常见的插件能力包括：

- Slash Commands
- Agents
- Skills
- Hooks
- MCP 配置
- 本地工具或语言服务接入
## 什么时候该用插件

### 你想把能力跨项目复用

如果某套流程会在多个项目里重复出现，例如代码审查、提交规范、前端设计检查、语言服务接入，插件比一次次复制配置更稳。

### 你想让团队共享同一套工作方式

当团队需要统一命令、统一质量门槛或统一外部工具接法时，插件会比聊天里口头约定更可靠。

### 你需要版本化和可维护性

如果一套能力会长期演进，插件的安装、更新和配置都更适合持续维护。

## 什么时候先别急着做插件

### 只是当前项目里的小实验

如果你只是试几条命令、临时挂一个 Skill，先用项目内配置会更轻。

### 规则还没有稳定

插件更适合已经成型的流程。还在频繁变化的阶段，先把能力跑顺，比过早封装更重要。

## 插件的典型生命周期

- 先明确你要解决的问题
- 选择现成插件、团队插件或本地插件
- 安装后先做小任务验证
- 再决定是否加入日常工作流
- 随着需求稳定，再做配置和版本治理
## 安装后先验证什么

```bash
/plugin list
```

然后至少检查 4 件事：

- 你需要的命令是否真的出现了
- 对话里是否能触发预期的 Agents 或 Skills
- 依赖的 MCP、语言服务或本地工具是否已经准备好
- 这套能力是否确实让你的常见任务更快，而不是更重
## 什么时候该自己做插件

如果你的需求明显带有团队特征，例如代码规范、提交流程、审查清单、内部工具入口或特定业务操作，自己做插件通常比硬套通用插件更合适。

这时建议你把目标收敛到一句话：

“装上这个插件后，团队在哪一类任务上能明显更稳、更快？”

答案越清晰，插件就越容易做对。
