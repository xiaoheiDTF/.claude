# 按用途选择插件

面对官方插件目录时，最容易浪费时间的方式不是“装错一个插件”，而是把插件当成一份长名单逐个浏览。更高效的方式，是先按你要解决的问题来选。

这页不是重复目录列表，而是把官方插件目录重组为几条更适合真实工作的选择路径。核心问题只有一个：

- 你现在最想改善的是哪一类工作？
如果你还没建立插件的基本心智模型，先看 [先理解插件](overview/)。如果你已经知道要从官方目录开始，但不知道先装哪几个，就从这页往下选。

## 一开始先把环境和自动化建议补齐

这类插件最适合“刚开始认真配置 Claude Code，但还不知道最该先上哪些自动化”的团队。

优先看：

- claude-code-setup
- hookify
- claude-md-management
### 它们分别解决什么

- claude-code-setup：先分析代码库，再建议最值得优先引入的 hooks、skills、subagents、commands 和 MCP 连接方式
- hookify：把“我不想再犯这个错误”快速沉淀成 hook 规则，而不是手写复杂配置
- claude-md-management：帮助你维护项目级 CLAUDE.md 和记忆入口，减少项目事实失真
### 什么时候先从这一组开始

- 团队还没有统一的 .claude/ 结构
- 不确定哪些自动化值得先做
- 想先把项目记忆、提醒和基础护栏搭起来
## 先把审查和质量门禁做好

如果你的问题已经不是“会不会用”，而是“怎样减少低质量改动进入主干”，优先看这一组。

优先看：

- code-review
- pr-review-toolkit
### 这组更适合什么

- PR 改动较大，需要多视角审查
- 团队已经有一定规范，希望把它转成更稳定的 review 流程
- 想减少“看起来有问题但其实是误报”的噪音
### 两者怎么区分

- code-review 更像一条标准化总审查链路：多代理并行、置信度过滤、结合 CLAUDE.md 规则检查
- pr-review-toolkit 更像一组可组合的审查角色：测试、错误处理、类型设计、简化代码、注释准确性等可以按需挑选
### 一个简单选法

- 想先有一条统一的 /code-review 工作流：先装 code-review
- 想把审查拆成多种专用视角：再补 pr-review-toolkit
## 把开发流程做成更稳定的工作路径

如果你的重点是“需求怎么推进得更稳”，而不是“只会多几个命令”，更应该看这组。

优先看：

- feature-dev
- commit-commands
### 它们最值得借鉴什么

- feature-dev：把需求澄清、代码库探索、架构权衡、实现和复查串成一条完整开发路径
- commit-commands：把提交与变更描述这类高频动作变得更一致
### 什么时候值得装

- 团队经常直接开写，缺少前置澄清和设计阶段
- 需求复杂、影响范围大，想减少“返工后才发现方向偏了”
- 想把提交信息、提交流程和开发节奏做得更整齐
## 补齐语言智能和本地工具能力

如果你已经明确知道自己需要的是“语言服务能力”，最有效的做法通常不是先找复杂工作流插件，而是先把语言智能补齐。

官方目录里这类插件通常包括：

- typescript-lsp
- pyright-lsp
- rust-analyzer-lsp
- gopls-lsp
- clangd-lsp
- ruby-lsp
- php-lsp
- swift-lsp
- kotlin-lsp
- csharp-lsp
- jdtls-lsp
- lua-lsp
### 这类插件适合什么场景

- 想增强 go-to-definition、引用查找、错误检查这类代码智能
- 多语言仓库里，不同语言都需要稳定的本地分析能力
- 团队希望把语言级反馈前置，而不是完全依赖运行后报错
### 这类插件的关键判断标准

- 本机是否已安装对应语言服务器或运行时
- 团队实际主力语言是什么
- 你需要的是“编辑期智能”还是“流程型自动化”
先把语言智能装对，往往比先加一堆复杂流程插件更能直接改善日常体验。

## 把设计构建和输出风格做得更可控

如果你的重点是前端质量、表达风格或解释方式，而不是底层工程自动化，这组更合适。

优先看：

- frontend-design
- explanatory-output-style
- learning-output-style
- playground
### 这组更适合什么

- 你希望 Claude 生成的前端界面更有明确审美方向
- 你需要更适合教学、解释或学习型输出的表达方式
- 你想在正式接入前先快速试验某种交互或输出风格
### 一个常见误区

这类插件解决的是“输出质量与表达方式”，不是项目治理。它们很有价值，但不应该替代 `rules/`、审查链路和团队配置。

## 搭建自己的插件或 MCP 能力

如果你已经确认现成能力不够，且准备把能力沉淀成团队资产，就进入这组。

优先看：

- plugin-dev
- mcp-server-dev
- skill-creator
- agent-sdk-dev
### 这组分别解决什么

- plugin-dev：帮助你构建和打包 Claude Code 插件本身
- mcp-server-dev：帮助你设计和实现 MCP server，并选择 remote HTTP、MCP app、MCPB 或本地 stdio 路径
- skill-creator：帮助你创建、改进和评估 Skill
- agent-sdk-dev：更适合进入 Agent SDK 相关的工程实现
### 什么时候该进入这组

- 官方插件已经不够贴合团队工作流
- 你需要接内部系统、私有服务或特定审批链
- 你准备把临时做法升级成可安装、可维护、可版本化的能力
## 推荐的最小组合

### 刚开始搭团队能力

- claude-code-setup
- hookify
- code-review
### 已经有流程，想把质量做稳

- code-review
- pr-review-toolkit
- 1 个与主力语言对应的 LSP 插件
### 准备沉淀为团队扩展

- plugin-dev
- mcp-server-dev
- skill-creator
## 一个简单判断原则

如果你当前的问题是“团队不知道先从哪种自动化开始”，先看环境与自动化建议。

如果你当前的问题是“改动质量不稳定”，先看审查与质量门禁。

如果你当前的问题是“需求推进经常跑偏”，先看开发流程与执行辅助。

如果你当前的问题是“编辑体验和静态分析不够强”，先看语言智能与本地工具。

如果你当前的问题是“现成能力不够贴合团队”，再进入插件/MCP 构建。
