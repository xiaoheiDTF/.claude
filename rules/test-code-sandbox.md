# test-code 沙盒规则

> 本规则约束辅助性脚本和工具的存放位置，并声明 `test-code` 为不受项目规范限制的沙盒目录。

---

## 1. 适用范围

- `doc/test-code/` 目录（由 `.gitignore` 单独保留跟踪）
- 所有临时性、辅助性、工具性代码

## 2. 存放限制

### 2.1 强约束（必须遵守）

| 文件类型 | 允许位置 | 说明 |
|---------|---------|------|
| `find_dup.py` | **仅** `doc/test-code/` | 重复代码查找脚本 |
| 其他辅助性 Python 脚本 | **仅** `doc/test-code/` | 如代码分析、统计、生成工具等 |
| 临时调试脚本 | **仅** `doc/test-code/` | 任何非业务代码的脚本 |

### 2.2 禁止行为

- **禁止** 在 `travel-agent/`、`travel-agent-java/`、`travel-web/` 等业务目录中创建 `find_dup.py` 或类似辅助脚本
- **禁止** 在业务代码目录中生成临时性、辅助性 Python 代码
- **禁止** 将辅助脚本混入源代码树

## 3. 沙盒特权

`doc/test-code/` 是一个**沙盒目录**，在此目录中：

- ✅ 可以写入任何编程语言的代码（Python、Shell、JavaScript 等）
- ✅ 不受项目正常代码规范限制（无需严格遵循 PEP 8、代码风格、类型注解等）
- ✅ 可以包含硬编码、魔法值、临时配置
- ✅ 可以依赖任何第三方库（只需在本地运行即可）
- ✅ 不需要写单元测试
- ❌ 但**不得**引用或导入业务代码（避免循环依赖）

## 4. 与业务代码的边界

```
doc/test-code/          ← 沙盒，任意代码
├── find_dup.py
├── chat_qwen.py
└── ...

travel-agent/           ← 业务代码，严格规范
travel-agent-java/      ← 业务代码，严格规范
travel-web/             ← 业务代码，严格规范
```

## 5. 为什么这样设计

1. **防止污染**：辅助脚本如果散落在业务目录中，会被误认为是正式代码，增加维护成本
2. **明确边界**：`test-code` 的沙盒身份让开发者清楚这里放的是"用完即走"的工具
3. **Git 管理**：通过 `.gitignore` 的例外规则，只保留 `doc/test-code/` 而忽略其他文档，避免仓库膨胀
